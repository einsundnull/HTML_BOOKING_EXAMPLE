i18n — Internationalisierung

Regeln für alle sichtbaren UI-Texte. Jeder Text ist von Anfang an i18n-kompatibel — Standardsprache Deutsch. Namespace entspricht immer dem Namen der aufrufenden HTML-Seite. Diese Datei ersetzt die i18n-Abschnitte aus Guideline_Layout_and_Styling und GUIDELINES_JS.

🟢 Arch-Rule 🔵 Soft-Rule ⛔ Verboten Standard: Deutsch (de)

1. Grundregeln

4 Regeln
🟢 Arch
Jeder für den User sichtbare Text ist i18n-kompatibel — von Anfang an
Kein hardcodierter Anzeigetext im HTML oder JS. Buttons, Labels, Fehlermeldungen, Platzhalter, ARIA-Labels, Tooltips — alles geht über i18n-Keys. Kein Text darf direkt in HTML-Attributen oder JS-Strings stehen.
❌ <button>Speichern</button> /* hardcoded */ ❌ btn.textContent = 'Speichern'; /* hardcoded in JS */ ✅ <button data-i18n="saveBtn">Speichern</button> ✅ btn.textContent = t('saveBtn'); /* über i18n-Funktion */
Layout · GUIDELINES_JS
🟢 Arch
Standardsprache: Deutsch (de)
Alle Fallback-Texte und Default-Werte sind auf Deutsch. Der HTML-Root trägt lang="de". Andere Sprachen werden als zusätzliche Übersetzungsdateien angelegt — Deutsch ist immer die primäre Quelle.
<html lang="de"> <!-- Alle Basis-Texte in Deutsch verfasst -->
Layout · GUIDELINES_JS
🟢 Arch
Namespace = Name der HTML-Seite ohne Extension
Die JSON-Übersetzungsdatei trägt denselben Namen wie die HTML-Seite die sie verwendet. Kein zentrales translations.json für alle Seiten.
HTML-SeiteLocale-DateiNamespace
student.htmllocales/student.jsonstudent
skiing-catalog.htmllocales/skiing-catalog.jsonskiing-catalog
wallet.htmllocales/wallet.jsonwallet
landing.htmllocales/landing.jsonlanding
Layout · GUIDELINES_JS
🟢 Arch
Dialoge und Overlays: Namespace der aufrufenden Seite
Ein Dialog der von student.html geöffnet wird, nutzt den Namespace student — auch wenn der Dialog eine eigene HTML-Datei in /dialogs/ ist. Der Namespace-Kontext bleibt immer bei der aufrufenden Seite.
/* student.html öffnet einen Dialog */ ✅ Namespace: "student" → locales/student.json ❌ Namespace: "dialog" → locales/dialog.json /* falsch */
Layout · GUIDELINES_JS

2. JSON-Struktur

3 Regeln
🟢 Arch
Hierarchie so flach wie möglich — maximal eine Ebene
Flat keys sind einfacher zu warten und schneller zu debuggen. Wenn eine zweite Ebene unvermeidbar ist, muss sie dokumentiert und begründet werden.
/* ✅ Richtig: eine Ebene */ { "pageTitle": "Lehrerkatalog", "filterBtn": "Filter", "filterGroupDiscipline": "Disziplin", "disciplineSki": "Ski", "expandBtn": "Details anzeigen", "collapseBtn": "Schließen", "errorNoResults": "Keine Lehrer gefunden" } /* ❌ Falsch: tief verschachtelt */ { "filter": { "group": { "discipline": "Disziplin" /* zu tief */ } } }

🟢 Arch
Key-Naming: camelCase, beschreibend, kontextuell
Keys beschreiben den Kontext (wo) + den Inhalt (was). Kein Mischen von Konventionen innerhalb einer Datei.

3. HTML-Attribute für i18n

1 Regel
🟢 Arch
data-i18n-Attribute für alle Text-Typen
Vier verschiedene Attribute je nach Verwendungszweck. Der Default-Text im HTML bleibt als Fallback erhalten.
<!-- Text Content --> <p data-i18n="hero.sub">Fallback-Text auf Deutsch</p> <!-- HTML Content (z.B. mit <br>, <strong>) --> <h1 data-i18n-html="hero.headline">Überschrift</h1> <!-- ARIA-Label (für Screen Reader, nicht sichtbar) --> <button aria-label="" data-i18n-aria="drawer.close">✕</button> <!-- Placeholder für Inputs --> <input placeholder="Name" data-i18n-placeholder="field.namePlaceholder">

5. Verbotene Muster

3 Einträge
⛔ Verboten
Hardcodierter Anzeigetext in HTML oder JS
Kein <button>Speichern</button> ohne data-i18n. Kein el.textContent = 'Gespeichert' direkt in JS.
Layout · GUIDELINES_JS
⛔ Verboten
Zentrales translations.json für alle Seiten
Führt zu nicht zuordenbaren Keys und Wartbarkeitsproblemen. Jede Seite hat ihre eigene Locale-Datei.
Layout
⛔ Verboten
Mehr als 2 JSON-Ebenen in Locale-Dateien
Tief verschachtelte Strukturen erhöhen die Fehleranfälligkeit und erschweren die Suche nach Keys.
Layout · GUIDELINES_JS
