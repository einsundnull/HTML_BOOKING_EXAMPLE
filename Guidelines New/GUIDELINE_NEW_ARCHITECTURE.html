Guideline · Architektur

Architektur

Strukturelle und technische Regeln des Projekts. Das System ist derzeit als Mockup im Browser (localStorage) implementiert und so angelegt, dass der Wechsel auf Firestore oder eine andere Datenbank mit minimalen Code-Änderungen an einer einzigen Stelle erfolgt. Diese Architektur-Entscheidungen sind nicht verhandelbar.

🟢 Arch-Rule 🔵 Soft-Rule ⛔ Verboten Stand: März 2026

Mockup-Strategie & Migrationspfad

Kernversprechen der Architektur: Wechsel von localStorage auf Firestore (oder jede andere Datenbank) = eine einzige Zeile Code-Änderung in app-config.js. Kein Consumer-Code wird dabei angefasst.
Schichten-Architektur
Consumer / UI-Layer — teacher.js · student.js · admin.js · skiing-catalog.js …
Ruft ausschließlich AppService.* auf. Kennt keine DB-Details. Ändert sich bei Migration nie.
↓ AppService.method(params, callback)
AppService — app-service.js
Definiert die API. Enthält keine Implementierungslogik. Delegiert an aktiven Adapter.
↓ delegates to AppAdapter (= eine Zeile in app-config.js)
LocalStorageAdapter (Mockup)
Wraps Store.* + ProfileStore.* · Synchron, aber Callback-Interface ist async-kompatibel.
FirestoreAdapter (Produktion)
Identische Signaturen. Aktivieren = eine Zeile in app-config.js auskommentieren.
↓ nur Adapter darf direkt zugreifen
Store / ProfileStore / ChatStore — store.js · profile.js · chat.js
Persistenz via localStorage. Wird bei Produktion durch Firestore-Collections ersetzt.
Migrations-Checkliste (wenn es soweit ist): Firebase SDK einbinden · adapter-firestore.js vollständig implementieren · Seed-Daten importieren · app-config.js: eine Zeile ändern · End-to-End-Test aller Consumer-Seiten · adapter-localstorage.js + catalog-seed.js entfernen. Kein Consumer-Code wird angefasst.

1. No-Build-Strategie

4 Regeln
Strategische Entscheidung: Kein Webpack, kein Vite, kein Babel, kein Transpiler. Maximale Browser-Kompatibilität ohne Build-Pipeline. Diese Entscheidung zieht alle folgenden Syntax-Regeln nach sich.
🟢 Arch
ES5-Syntax — var, function(){}, String-Konkatenation
Keine modernen Syntax-Features ohne Transpiler. Begründung: Direktausführung im Browser ohne Build-Schritt.
RichtigVerboten
var x = 1;let x · const x
function foo() {}() => {}
'Hallo ' + name`Hallo ${name}`
if (x == null)x?.y · x ?? y
x['key']{ key } (Destructuring)
Array.prototype.concat.apply([], arr)...arr (Spread)
GUIDELINES_JS
🟢 Arch
Kein import / export — IIFE-Pattern für alle Module
Module kapseln ihren Scope via IIFE und exponieren ein öffentliches Interface als globale Variable. Kein globaler Scope-Pollution.
/* Jedes .js-Modul folgt diesem Pattern */ var MyModule = (function() { 'use strict'; var _private = null; function _privateMethod() { /* intern */ } function publicMethod() { /* extern */ } return { publicMethod: publicMethod }; }());
GUIDELINES_JS · STYLE-GUIDE
🟢 Arch
HTML, CSS und JS immer in getrennten Dateien
Keine <style>-Tags im HTML-Body. Keine <script>-Blöcke außerhalb von <head>. Ausnahme: <template>-Elemente in HTML.
STYLE-GUIDE · Layout
🟢 Arch
HTML-Escaping bei jedem User-Daten-Output in innerHTML
Die _esc(str)-Funktion ist Pflicht bei jedem String, der aus User-Daten stammt und in innerHTML landet. XSS-Schutz ohne Framework.
function _esc(str) { return String(str || '') .replace(/&/g, '&amp;') .replace(/</g, '&lt;') .replace(/>/g, '&gt;') .replace(/"/g, '&quot;') .replace(/'/g, '&#39;'); }

2. Adapter-Pattern & Datenzugriff

4 Regeln
🟢 Arch
Consumer-Code greift NIEMALS direkt auf Store.* / localStorage zu
Der einzige erlaubte Zugriffspunkt für Daten im Consumer-Code ist AppService.*. Store.* und ProfileStore.* darf ausschließlich adapter-localstorage.js aufrufen.
✅ AppService.getTeachersWithProfiles(function(err, teachers) { if (err) { Toast.error(err.message); return; } /* teachers verwenden */ }); ❌ var teachers = Store.Users.byRole('teacher'); /* verboten */ ❌ var profile = ProfileStore.getOrDefault(uid); /* verboten */ ❌ var raw = JSON.parse(localStorage.getItem('x')); /* verboten */
Guideline_Architecture · GUIDELINES_JS
🟢 Arch
Alles ist async — function(err, result) überall, ausnahmslos
Auch wenn LocalStorageAdapter intern synchron arbeitet — der Consumer-Code verlässt sich nie auf Synchronizität. Beim Wechsel auf Firestore sind die Callbacks dann asynchron, ohne dass Consumer-Code geändert wird.
/* Richtig: immer Node.js-Callback-Konvention */ AppService.getUser(uid, function(err, user) { if (err) { Toast.error('Fehler: ' + err.message); return; } /* user sicher verwenden */ }); /* Erfolgs-Callback */ callback(null, result); /* Fehler-Callback */ callback(new Error('Beschreibung'), null);
GUIDELINES_JS
🟢 Arch
Der Adapter-Swap: eine Zeile in app-config.js
Das ist der vollständige Migrations-Aufwand auf Code-Ebene. Kein Suchen & Ersetzen. Kein Consumer-Refactoring.
/* app-config.js — einzige Datei die das Backend bestimmt */ var APP_VERSION = '2026-03-19-1'; /* MOCKUP (Entwicklung) */ var AppAdapter = LocalStorageAdapter; /* PRODUKTION — einfach diese eine Zeile auskommentieren: */ /* var AppAdapter = FirestoreAdapter; */
Guideline_Architecture · GUIDELINES_JS
🟢 Arch
Fehler niemals verschlucken — Adapter muss immer callback(err, null) liefern
Kein stiller Fail. Wenn eine Operation fehlschlägt, ist es Aufgabe des Consumers, den Fehler sichtbar zu machen (Toast, Modal oder Fehler-Element im Dialog).

3. Datenschema — Firestore-kompatibel

4 Regeln
Warum Firestore-kompatibel von Anfang an? Wenn das Schema heute sauber modelliert ist, kann Firestore morgen als Drop-in genutzt werden. Jede localStorage-Collection entspricht direkt einer Firestore-Collection.
🟢 Arch
Flache Arrays als Collections — kein tief verschachteltes JSON
Max. 1 Ebene Verschachtelung. Collections sind Arrays von flachen Objekten — genau wie Firestore-Collections. Keine profile.teaching.levels.beginner.active-Strukturen.
RichtigVerboten
[{uid, name, role}, …]{uid: {name, role}, …}
languages: ['de', 'en']language_de: true, language_en: true
Codes: 'de', 'beginner'Klartexte: 'Deutsch', 'Anfänger'
GUIDELINES_JS
🟢 Arch
Bestehende Schemas sind unveränderlich
User · Slot · Recurring · Selection · Profile — diese Schemas dürfen nicht umbenannt oder restrukturiert werden. Neue Felder dürfen addiert werden (additive change), bestehende Felder nicht umbenannt (breaking change).
/* Unveränderliche Kern-Schemas */ User: { uid, name, role, email, password, discipline } Slot: { slotId, teacherId, studentId|null, date, time, status, baseStatus } Recurring: { recurringId, teacherId, dayOfWeek, time } Selection: { studentId, teacherId } Wallet: { uid, balance, currency, updatedAt } Tx: { txId, uid, type, amount, balance, description, status, createdAt, relatedUid }
GUIDELINES_JS
🔵 Soft
Neue Entitäten VOR Implementierung in CRUD_SERVICE_REFERENCE_v2.md dokumentieren
Kein wildes Hinzufügen von Feldern oder Collections. Schema-Entscheidungen zuerst dokumentieren, dann implementieren.
GUIDELINES_JS
🟢 Arch
Seed-Daten isoliert in Seed-Dateien — nie in Consumer- oder Store-Code
Testdaten gehören ausschließlich in catalog-seed.js oder gleichwertige Feature-Seed-Dateien. Seed prüft immer zuerst ob Daten existieren.
AppService.getTeachers(function(err, teachers) { if (err || teachers.length > 0) return; /* bereits geseeded */ /* Seed-Daten einfügen */ });

4. Ladereihenfolge & Asset-Versioning

5 Regeln
🟢 Arch
Feste Script-Ladereihenfolge — app-config.js immer zuerst
app-config.js definiert APP_VERSION und AppAdapter. Kein anderes Script darf geladen werden bevor app-config.js verfügbar ist.
<!-- Pflicht-Reihenfolge für jede HTML-Seite --> <script src="js/app-config.js"></script> <!-- 0. IMMER ZUERST --> <script src="js/store.js?v=APP_VER"></script> <!-- 1. --> <script src="js/auth.js?v=APP_VER"></script> <!-- 2. --> <script src="js/profile.js?v=APP_VER"></script> <!-- 3. --> <script src="js/ui.js?v=APP_VER"></script> <!-- 4. --> <script src="js/navbar.js?v=APP_VER"></script> <!-- 5. --> <script src="js/adapter-localstorage.js?v=APP_VER"></script> <!-- 6. --> <script src="js/app-service.js?v=APP_VER"></script> <!-- 7. --> <script src="js/[seite].js?v=APP_VER"></script> <!-- 8. Seiten-spezifisch -->
Guideline_Architecture · Cache-Guideline
🟢 Arch
Assets werden ausschließlich über app-config.js injiziert — niemals hardcoded in HTML
app-config.js enthält die Asset-Listen pro Seite (_PAGE_ASSETS) und injiziert alle <link> und <script> Tags dynamisch mit APP_VERSION. Jede HTML-Seite enthält im <head> ausschließlich: Meta-Tags, Google-Fonts-Link, und <script src="./js/app-config.js"></script> ohne ?v=. Wenn neue CSS- oder JS-Dateien hinzukommen, werden sie in _PAGE_ASSETS in app-config.js eingetragen.
/* ❌ Verboten — hardcoded ?v= in HTML: */ <link rel="stylesheet" href="./css/tokens.css?v=2026-03-19-1"> <script src="./js/student.js?v=2026-03-19-1"></script> /* ✅ Korrekt — nur app-config.js in HTML: */ <script src="./js/app-config.js"></script> /* ✅ Assets in app-config.js/_PAGE_ASSETS pflegen: */ 'student.html': { css: ['./css/tokens.css', './css/student.css', ...], js: ['./js/student.js', ...] }
Cache-Guideline · aktualisiert 2026-03-19
🟢 Arch
APP_VERSION bei JEDER inhaltlichen Änderung an .js oder .css erhöhen
Format: YYYY-MM-DD-{increment}. Zwei Änderungen am selben Tag: 2026-03-19-1 → 2026-03-19-2. APP_VERSION definiert ausschließlich in app-config.js (SSOT). Durch _injectAssets() gilt die neue Version automatisch für alle Seiten — kein manuelles Suchen und Ersetzen in HTML-Dateien.
Cache-Guideline
🟢 Arch
CSS-Ladereihenfolge: tokens → base → components → navbar → [page-specific]
Token-Verfügbarkeit muss für alle nachfolgenden Stylesheets garantiert sein. login.css nur auf Seiten die AuthModal oder Google-Picker nutzen.
Layout
⛔ Verboten
"Bitte Cache leeren" als Workaround
Wenn Nutzer oder Entwickler Cache leeren müssen, ist das ein Architektur-Fehler. Lösung: APP_VERSION erhöhen. Kein Hinweis in README, kein Hinweis in Commit-Messages.

6. Verbotene Muster

4 Einträge
⛔ Verboten
Direkter Store.* / localStorage-Zugriff im Consumer-Code
Macht die Firestore-Migration unmöglich. Immer AppService.* nutzen.
GUIDELINES_JS
⛔ Verboten
Asset-Referenzen ohne ?v=...
Browser cached alte Datei. Gilt für JS, CSS und alle statischen Ressourcen.
Cache-Guideline
⛔ Verboten
Seed-Daten in Consumer- oder Store-Dateien
Testdaten gehören isoliert in Seed-Dateien. Kein Test-Code in Produktions-Code.
GUIDELINES_JS
⛔ Verboten
Tief verschachtelte Objekte im Datenschema (>1 Ebene)
Nicht Firestore-kompatibel. Flache Strukturen, Array-Felder, Codes statt Klartexte.
