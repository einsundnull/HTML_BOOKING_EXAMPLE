Guideline · Dialoge & Overlays

Dialoge & Overlays

Regeln für alle modalen Fenster, Drawer, Alerts und Overlays. Browser-Dialoge sind verboten. Jeder Dialog ist ein eigenständiges Objekt mit sichtbarer Fehlerbehandlung. Diese Datei konsolidiert §5 aus Guideline_Layout_and_Styling und §10 aus GUIDELINES_JS.

🔴 Hard-Rule 🟢 Arch-Rule ⛔ Verboten Stand: März 2026

1. Kernregeln

4 Regeln
Keine Browser-Dialoge. window.alert(), window.confirm() und window.prompt() sind ohne Ausnahme verboten. Sie sind nicht themebar, blockieren den Rendering-Thread und entsprechen nicht dem Projekt-Styling.
🔴 Hard
Alle Dialoge und Overlays entsprechen dem Styling der Hauptseite
Tokens, Komponenten-Klassen, Schrift — alles folgt dem Projekt-Design. Kein generisches Browser-Styling, keine abweichenden Farben oder Fonts.
/* Dialog nutzt Projekt-Tokens */ .dialog { background: var(--bg-surface); border: 1px solid var(--border); border-radius: var(--radius-lg); box-shadow: var(--shadow-md); font-family: var(--font); padding: var(--sp-4); }
Layout · GUIDELINES_JS
🟢 Arch
Dialoge und Overlays sind immer eigenständige Objekte (IIFE oder Object-Literal)
Kein Inline-Dialog-Code in der Hauptseite. Jeder Dialog kapselt seinen Zustand und seine Logik.
/* Jeder Dialog ist ein eigenes Objekt */ var ConfirmDialog = (function() { 'use strict'; var _overlay = null; function show(opts) { /* title, message, onConfirm, onCancel */ } function hide() { } function _showError(context, err) { /* PFLICHT */ } return { show: show, hide: hide }; }());
Layout · GUIDELINES_JS
🟢 Arch
Standalone-Dialoge als eigene HTML-Dateien in /dialogs/
Komplexe Dialoge (Filterschublade, komplexe Formulare) als eigenständige .html-Datei in /dialogs/. Sie laden app-config.js als erstes Script und versionieren alle Assets mit ?v=${APP_VERSION}.
HTML/ └── dialogs/ ├── catalog-filter-drawer.html ← eigenständig ├── confirm-dialog.html └── booking-dialog.html Jede Dialog-HTML lädt zuerst app-config.js: <script src="../js/app-config.js"></script> <script src="../js/ui.js?v=..."></script>
Layout · GUIDELINES_JS
🔴 Hard
ESC-Taste + Klick außerhalb schließt jeden Dialog
Kein Dialog ohne diese beiden Schließmechanismen. Zusätzlich immer ein sichtbarer Schließen-Button (X oder "Abbrechen").
/* Pflicht-Handler für jeden Dialog */ document.addEventListener('keydown', function(e) { if (e.key === 'Escape') { Dialog.hide(); } }); overlay.addEventListener('click', function(e) { if (e.target === overlay) { Dialog.hide(); } });
Layout

2. Fehlerbehandlung in Overlays

3 Regeln
Das größte Debugging-Problem bei Overlays: Fehler die im Overlay-Code auftreten sind in der Browser-Konsole oft schwer zu finden oder bleiben ganz unsichtbar. Jeder Overlay-Fehler muss im UI sichtbar gemacht werden.
🔴 Hard
Jeder Dialog implementiert _showError(context, err)
Pflicht-Methode in jedem Dialog-Objekt. Zeigt den Fehler als sichtbares Element — nicht nur in der Konsole. Minimale Implementierung: Fehler via Toast.error() anzeigen. Bevorzugte Implementierung: Fehler-Element innerhalb des Dialogs einblenden.
/* Pflicht: _showError in jedem Dialog */ function _showError(context, err) { var msg = '[' + context + '] ' + (err && err.message ? err.message : String(err)); /* Option A: Toast (Minimum) */ Toast.error(msg); /* Option B: Fehler-Element im Dialog (bevorzugt) */ var errEl = document.getElementById('dialog-error'); if (errEl) { errEl.textContent = msg; errEl.classList.remove('is-hidden'); } }
GUIDELINES_JS · Layout
🔴 Hard
Jede asynchrone Overlay-Operation in try/catch mit _showError
Kein stiller Fail. AppService-Callbacks immer auf Fehler prüfen und _showError aufrufen.
function _loadData() { try { AppService.getTeachers(function(err, teachers) { if (err) { _showError('_loadData', err); return; } _render(teachers); }); } catch(e) { _showError('_loadData:catch', e); } } /* ❌ Verboten: Fehler verschlucken */ AppService.getTeachers(function(err, data) { if (err) return; /* err wird ignoriert — verboten */ });
GUIDELINES_JS · Guideline_Architecture
🔴 Hard
Fehler-Element im Dialog-HTML als eigenes Element
Jedes Dialog-HTML enthält ein vorbereitetes Fehler-Element das standardmäßig verborgen ist und bei Bedarf eingeblendet wird.
<!-- Pflicht-Element in jedem Dialog-HTML --> <div id="dialog-error" class="dialog-error-msg is-hidden" role="alert" aria-live="polite" ></div> /* CSS */ .dialog-error-msg { background: var(--hard-bg); border: 1px solid var(--hard-border); border-radius: var(--radius-sm); color: var(--hard-text); font-size: var(--text-small); padding: var(--sp-2) var(--sp-3); margin-top: var(--sp-2); }
Layout · GUIDELINES_JS

3. Standard-Dialog-Template

Modal.show() aus ui.js ist der Standard-Weg für einfache Bestätigungs- und Informationsdialoge. Nur für komplexe Dialoge (Formulare, Filterschubladen) wird ein eigenständiges Dialog-Objekt erstellt.
/* Einfacher Dialog via Modal.show() (ui.js) */ var dialog = Modal.show({ title: 'Buchung stornieren', bodyHTML: '<p>Möchtest du diese Buchung wirklich stornieren?</p>', footerHTML: '<button class="btn btn-ghost" data-action="cancel">Abbrechen</button>' + '<button class="btn btn-danger" data-action="confirm">Stornieren</button>' }); /* Dialog-HTML-Struktur (für standalone Dialoge) */ <div class="modal-overlay" role="dialog" aria-modal="true" aria-labelledby="modal-title"> <div class="modal"> <div class="modal-header"> <h2 id="modal-title" data-i18n="dialogTitle"></h2> <button class="btn btn-ghost" id="modal-close" aria-label="Schließen" data-i18n-aria="dialogClose">✕</button> </div> <div class="modal-body"> <div id="dialog-error" class="dialog-error-msg is-hidden" role="alert" aria-live="polite"></div> <!-- Inhalt --> </div> <div class="modal-footer"> <button class="btn btn-ghost" data-i18n="cancelBtn">Abbrechen</button> <button class="btn btn-primary" data-i18n="confirmBtn">Bestätigen</button> </div> </div> </div>

4. Accessibility-Anforderungen

3 Regeln
🔴 Hard
ARIA-Attribute: role="dialog", aria-modal="true", aria-labelledby
Jeder Dialog-Wrapper trägt diese drei Attribute. Das Fehlen macht den Dialog für Screen Reader unsichtbar.
STYLE-GUIDE
🔴 Hard
Focus-Management: Focus geht beim Öffnen in den Dialog, beim Schließen zurück
Beim Öffnen: firstFocusable.focus(). Beim Schließen: triggerElement.focus(). Tab-Trap innerhalb des offenen Dialogs.
function show() { _overlay.classList.remove('is-hidden'); var firstBtn = _overlay.querySelector('button, input, [tabindex="0"]'); if (firstBtn) { firstBtn.focus(); } } function hide() { _overlay.classList.add('is-hidden'); if (_trigger) { _trigger.focus(); } /* zurück zum Auslöser */ }
STYLE-GUIDE
🔴 Hard
Fehler-Element mit role="alert" und aria-live="polite"
Das Fehler-Element wird von Screen Readern automatisch vorgelesen wenn es eingeblendet wird — ohne dass der Focus wechseln muss.
STYLE-GUIDE · Layout

5. i18n in Dialogen

2 Regeln
🟢 Arch
Dialog-Texte im Namespace der aufrufenden Seite
Auch wenn ein Dialog eine eigene HTML-Datei hat — seine Texte leben in der locales/[aufrufende-seite].json. Kein separates dialog.json.
Aufrufende SeiteDialog-Texte in
student.htmllocales/student.json
skiing-catalog.htmllocales/skiing-catalog.json
teacher.htmllocales/teacher.json
Layout · GUIDELINES_JS
🟢 Arch
Dialog-Keys mit kontextuellem Prefix (dialogConfirm*, dialogBooking*…)
Dialog-spezifische Keys bekommen einen Prefix der sie von Seiten-Keys unterscheidet. Flache Hierarchie bleibt erhalten.
/* student.json — Dialog-Keys mit Prefix */ { "pageTitle": "Mein Dashboard", "dialogCancelTitle": "Buchung stornieren", "dialogCancelMessage": "Möchtest du wirklich stornieren?", "dialogCancelConfirmBtn": "Stornieren", "dialogCancelAbortBtn": "Abbrechen" }
Layout

6. Verbotene Muster

4 Einträge
⛔ Verboten
window.alert() / window.confirm() / window.prompt()
Nicht themebar, blockiert Rendering, nicht i18n-kompatibel. Immer Modal.show() oder eigenes Dialog-Objekt.
Layout
⛔ Verboten
Fehler nur in console.error() loggen — ohne sichtbare UI-Ausgabe
Console-Logs sind für Entwickler, nicht für Users. Jeder Fehler der im Dialog auftritt, muss auch im UI sichtbar werden.
GUIDELINES_JS · Layout
⛔ Verboten
Dialog-Code direkt in der Hauptseite inline
Dialog-Logik gehört in ein eigenständiges Objekt oder eine eigenständige Datei — nicht als Inline-Code in teacher.js, student.js etc.
GUIDELINES_JS
⛔ Verboten
Dialog ohne ESC-Schließung oder ohne Schließen-Button
Jeder Dialog muss via ESC-Taste, Klick auf Backdrop und einen sichtbaren Button schließbar sein.
Layout
