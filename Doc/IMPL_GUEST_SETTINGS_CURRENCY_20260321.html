GuestSettings — Währung via AppService-Layer

Store.GuestSettings war nie implementiert. Die Navbar-Währungsauswahl schrieb ins Leere — Lehrerpreise auf der Visitenkarte ignorierten die Einstellung. Jetzt vollständig über den Adapter-Layer implementiert: store.js → adapter-localstorage.js → app-service.js → currency-service.js.

🔴 Bugfix 🔵 IMPL 📅 21. März 2026 4 Dateien geändert ✅ APP_VERSION 2026-03-21-22
1 Root Cause — was fehlte store.js · currency-service.js
Kernproblem: currency-service.js rief Store.GuestSettings.setCurrency() und Store.GuestSettings.getCurrency() auf — aber Store.GuestSettings war in store.js nie definiert. Der typeof Store !== 'undefined' && Store.GuestSettings-Guard verhinderte den Crash, machte den Code aber zu totem Code. Kein Schreibvorgang. Kein Lesevorgang. Einstellung wurde nicht gespeichert.

4 Architektur-Entscheid — Sync-Lesezugriff in getUserCurrency() currency-service.js · Guideline §2
Warum liest getUserCurrency() noch direkt aus Store.GuestSettings?
getUserCurrency(uid) wird synchron aus Render-Pfaden aufgerufen (_buildCard(), _buildHeroSection(), etc.). Ein async-Callback wäre hier strukturell nicht integrierbar ohne alle Render-Funktionen umzuschreiben — was außerhalb des Scope dieser Implementierung liegt.
Operation Pfad Begründung Guideline-konform?
Schreiben
setGuestCurrency()		 navbar.js → CurrencyService → AppService → Adapter → Store.GuestSettings Write-Operation ist fire-and-forget — Callback-Pattern problemlos integrierbar ✅ Voll konform
Lesen
getUserCurrency()		 Render-Funktion → CurrencyService → Store.GuestSettings direkt Sync-Render-Pfad. GuestSettings ist ein einzelnes, flaches Objekt ohne relationale Abhängigkeiten — vergleichbar mit einem Config-Read. Beim Wechsel auf Firestore würde dieser Sync-Zugriff durch einen gecachten Wert ersetzt. ⚠️ Dokumentierte Ausnahme
Firestore-Migrationspfad: Beim Wechsel auf Firestore wird getUserCurrency() den GuestSettings-Wert aus einem lokalen Cache lesen, der beim Seitenstart via AppService.getGuestCurrency() befüllt wird — analog zu CurrencyService.warmUp() für den Rate-Cache. Consumer-Code wird nicht geändert.
