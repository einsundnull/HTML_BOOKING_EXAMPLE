Grouping-Logik — mergeBookingBlocks (Wiederverwendung)
ui.js → financial.js
Kein redundanter Code. mergeBookingBlocks() ist eine globale Funktion in ui.js — bereits in teacher.js und student.js verwendet. financial.js ruft diese Funktion auf; kein eigener Merge-Algorithmus notwendig.
financial.js — renderBookings() Grouping-Aufruf
/* Reuse mergeBookingBlocks from ui.js (global) */ var blocks; if (typeof mergeBookingBlocks === 'function') { blocks = mergeBookingBlocks(date, daySlots, _today, { groupField: 'studentId', // group consecutive slots per student resolveParty: function(slot) { return { studentId: slot.studentId }; } }); } else { /* Fallback: each slot is its own block */ blocks = daySlots.map(function(s) { return { start: s.time, end: AppService.slotEndTime(s.time), bookedSlots: [s], studentId: s.studentId, isFullyConfirmed: s.status === 'confirmed' }; }); }
Fallback-Sicherheit: Der typeof mergeBookingBlocks === 'function'-Check stellt sicher, dass die Seite auch dann funktioniert wenn ui.js nicht geladen ist (z.B. in Tests oder zukünftigen Umbau-Szenarien).
mergeBookingBlocks ParameterWert in financial.jsBedeutung
dateStrAktuelles Datum aus dateOrderWird in den Block-Output geschrieben
bookedSlotsAlle Slots des Tages nach Chip-Filter + DatumsfilterInput für Merge-Algorithmus
today_today (ISO-String)Für isFullyConfirmed Berechnung
opts.groupField'studentId'Konsekutive Slots werden nur gruppiert wenn selbe Student-ID
opts.resolvePartyslot → { studentId }Kopiert studentId in Block-Objekt

mergeBookingBlocks gibt Blöcke zurück mit: start, end, bookedSlots[], isFullyConfirmed, hasPending und alle Felder aus resolveParty(). Die Blockrenderer-Logik liest daraus Zeitspanne, Slot-Anzahl, Gesamtpreis und Bestätigungsstatus.

Datumsfilter-Architektur
financial.js — isolierter Zustand, kein globales bookingsFilter
Bewusste Design-Entscheidung: _buildSortDateRangeRow() aus ui.js wurde NICHT verwendet, da diese Funktion das globale bookingsFilter-Objekt liest/schreibt — das ist der Zustand von teacher.js / student.js. financial.js hat seinen eigenen isolierten Zustand und darf diesen nicht überschreiben.

Filter-Pipeline — Buchungsbereich
1
Chip-Filter (_bkFilter)
Alle, Unbestätigt, Bestätigt, Zukünftig, Vergangen, Deposit, Fällig, Refund. Setzt auch den _activeKpi-Zustand zurück bei manuellem Klick.
2
Datumsfilter (_bkDateFrom / _bkDateTo)
Vergleich slot.date >= _bkDateFrom (ISO-String-Vergleich, lexikographisch korrekt für YYYY-MM-DD). Nur aktiv wenn nicht leer. Kombination mit Chip-Filter möglich.
3
Gruppierung nach Datum
byDate{}-Map — Slots nach slot.date gebündelt. dateOrder[] nach neuesten zuerst sortiert.
4
mergeBookingBlocks — konsekutive Slots zusammenführen
Pro Tag werden consecutive Slots desselben Schülers zu einem Block zusammengefasst. Ausgabe: blocks[] mit start, end, bookedSlots[], isFullyConfirmed.
5
Rendering — ein <li> pro Block
Zeitspanne (block.start–block.end), Slot-Anzahl bei >1, Gesamtpreis des Blocks, Bestätigungs-Badge. KPI-Drill-down Extra-Info aus _escrowMap.
