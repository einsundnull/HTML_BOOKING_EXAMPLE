Profile View — Unbefugter Edit-Zugriff
Jeder Besucher eines Lehrerprofils sah den Edit-Button und den „Profil bearbeiten"-Link — unabhängig davon, ob er der Profilinhaber war. Root Cause: Unsicherer Auth.current()-Fallback in ProfileView.init() las die falsche ?uid= aus der URL.
🔴 Security · Kritisch 📅 21. März 2026 1 Datei geändert · profile.js ✅ Behoben · APP_VERSION 2026-03-21-5
1

Root Cause

Kern des Problems: Auth.current() liest ?uid= aus der aktuellen URL. Auf profile-view.html identifiziert ?uid= jedoch den Profilinhaber, nicht den eingeloggten Besucher. Der Fallback führte dazu, dass _viewer.uid === _uid immer true war — und der Edit-Button für jeden Besucher sichtbar wurde.
profile.js — ProfileView.init() — VORHER (fehlerhaft)
/* viewer = the logged-in user, passed explicitly so we never confuse the profile being viewed (?uid=) with the session owner */ var viewerUid = params.get('viewer'); _viewer = viewerUid ? AppService.getUserSync(decodeURIComponent(viewerUid)) : Auth.current(); /* BUG: wenn viewerUid = '' (leer) → Auth.current() aufgerufen Auth.current() liest ?uid= aus profile-view.html URL ?uid= = Profilinhaber → _viewer = Profilinhaber → _viewer.uid === _uid → true → Edit-Button für ALLE sichtbar */
profile.js — ProfileView.init() — NACHHER (behoben)
/* SECURITY: Auth.current() MUST NOT be used as fallback here. On profile-view.html the ?uid= param identifies the PROFILE being viewed — not the session owner. If Auth.current() were called it would return the profile owner, making _viewer.uid === _uid always true and granting edit access to every visitor. Rule: if ?viewer= is absent or empty → _viewer = null → read-only. */ var viewerUid = params.get('viewer'); _viewer = (viewerUid && viewerUid.trim()) ? AppService.getUserSync(decodeURIComponent(viewerUid.trim())) : null;

Exploit-Pfad — Schritt für Schritt

Browser Direktaufruf
Benutzer öffnet skiing-catalog.html direkt (kein ?uid= in der URL) oder ist nicht eingeloggt. Auth.current() auf der Katalogseite gibt null zurück.
↓
skiing-catalog.js _buildCard()
Link zum Profil wird mit leerem viewer gebaut:
profile-view.html?uid=t2&viewer=
(da Auth.current() = null → viewer='')
↓
profile.js ProfileView.init()
viewerUid = params.get('viewer') → '' (leer, aber nicht null)
'' → falsy → Fallback: _viewer = Auth.current()
↓
auth.js Auth.current()
Liest ?uid= aus der aktuellen URL (profile-view.html?uid=t2&viewer=)
→ Store.Users.byUid('t2') → gibt Profilinhaber T2 zurück
↓
profile.js _viewer.uid === _uid
_viewer.uid = 't2', _uid = 't2'
→ true → Edit-FAB sichtbar, „Profil bearbeiten"-Button sichtbar — für jeden Besucher
Wichtig: Der Bug tritt auch auf, wenn ein eingeloggter Schüler den Katalog über einen Direktlink öffnet (z.B. Browser-Lesezeichen auf skiing-catalog.html ohne ?uid=). Es ist kein anonymer Zugriff notwendig — der Schüler könnte über den fehlerhaft gesetzten Edit-Button auf profile-edit.html?uid=t2&owner=t2 navigieren. ProfileEdit prüft owner gegen Auth.current() und würde den Zugriff dort blockieren — aber die irreführende UI war trotzdem sichtbar.

5

Implementierter Fix

1 Datei · profile.js
Fix-Strategie: Der Fix liegt ausschliesslich in profile.js. Der Auth.current()-Fallback wird entfernt. Wenn ?viewer= fehlt oder leer ist, bleibt _viewer = null. Alle drei Ownership-Checks (_buildCTASection, _bindJumper, Z. 1811) schlagen dann fehl → kein Edit-Zugriff. Korrekte Aufrufer (navbar, student.js) sind nicht betroffen, da sie bereits einen validen viewer-Param übergeben.
