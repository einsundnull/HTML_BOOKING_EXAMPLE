2 Architektur

Die neue Buchungskarte folgt dem bestehenden Muster von tx_inquiry und booking_request: eine strukturierte type: 'service'-Nachricht mit eigenem serviceEvent: 'new_booking' und einem bookingSnapshot-Objekt mit allen relevanten Daten. Das Rendering erfolgt in _buildServiceMsg() in chat.js.

/* Nachrichtentyp-Hierarchie */
msg.type === 'service'               // service card (kein Avatar-Bubble)
  msg.serviceEvent === 'student_request'   // Schüleranfrage
  msg.serviceEvent === 'booking_request'   // Buchungsanfrage (teacher initiiert)
  msg.serviceEvent === 'new_booking'       // ← NEU: Buchungsbestätigung
  msg.serviceEvent === 'student_disconnected'
  msg.serviceEvent === 'tx_inquiry'

Die neue Karte rendert als eigene Wrapper-Klasse .chat-msg-booking-notify (zentriert, breiter als normale Service-Cards), mit Avatar des Senders, vollständigem Datum, allen Buchungsdetails und einem Tap-Hinweis. Ein Klick öffnet das WalletCore.showTxDetail()-Sheet.

4 ChatStore — sendNewBookingNotification()

Neue Funktion in ChatStore (chat.js). Erstellt eine type: 'service'-Nachricht mit serviceEvent: 'new_booking' und einem bookingSnapshot-Objekt.

function sendNewBookingNotification(senderId, receiverId, bookingData) {
  var msg = {
    msgId:           _uuid(),
    conversationKey: _conversationKey(senderId, receiverId),
    senderId:        senderId,
    receiverId:      receiverId,
    text:            '',
    type:            'service',
    isService:       true,
    serviceEvent:    'new_booking',
    teacherId:       bookingData.teacherId,
    studentId:       bookingData.studentId,
    bookingSnapshot: {
      dateLabel:      bookingData.dateLabel,      // z.B. "Do., 19. März 2026"
      dateStr:        bookingData.dateStr,        // ISO "2026-03-19" für TX-Lookup
      blockStart:     bookingData.blockStart,     // "14:00"
      blockEnd:       bookingData.blockEnd,       // "16:30"
      slotCount:      bookingData.slotCount,      // 5
      price:          bookingData.price,          // Gesamtpreis EUR
      depositAmount:  bookingData.depositAmount,
      depositType:    bookingData.depositType,
      depositPercent: bookingData.depositPercent,
      paymentMode:    bookingData.paymentMode,    // 'instant' | 'cash_on_site'
      initiatorRole:  bookingData.initiatorRole   // 'student' | 'teacher'
    },
    readStatus: 'sent', edited: false, deleted: false,
    createdAt:  _now()
  };
  // saved to localStorage via _save()
}

Die Funktion ist in ChatStore's public API exportiert und kann mit typeof ChatStore.sendNewBookingNotification === 'function' geprüft werden (graceful fallback auf plain text wenn nicht verfügbar).

9 Klick → TX-Detail Flow

Beim Klick auf die Buchungskarte läuft folgende Logik ab:

card.addEventListener('click', function() {
  // 1. Alle Transaktionen des aktuellen Users laden
  AppService.getTransactions(_currentUser.uid, function(err, txs) {

    // 2. Passendes TX suchen: type='booking' + slotDate + slotTimeStart + teacherId/studentId
    for (var i = 0; i < txs.length; i++) {
      var tx = txs[i];
      if (tx.type === 'booking' &&
          tx.meta.slotDate      === snap.dateStr &&
          tx.meta.slotTimeStart === snap.blockStart &&
          (tx.meta.teacherId === tchId || tx.meta.studentId === stuId)) {
        WalletCore.showTxDetail(tx, _currentUser.uid); // ✅ Echtes TX gefunden
        return;
      }
    }

    // 3. Fallback: Synthetic TX aus bookingSnapshot für Anzeige
    WalletCore.showTxDetail({ type: 'booking', meta: { ...snap } }, _currentUser.uid);
  });
});

Der Synthetic-TX-Fallback greift wenn das echte TX noch nicht im lokalen Store ist (z.B. async-Verzögerung) — die Anzeige ist dann vollständig, aber txId ist null.

12 Offene Punkte

Phase 2 — Empfehlungen
  • Teacher-initiierte Buchungen: Wenn ein Lehrer einen Schüler direkt einplant (teacher.js), sollte sendNewBookingNotification ebenfalls aufgerufen werden mit initiatorRole: 'teacher'
  • Recurring-Buchungen: Die neue _openRecurringBookingDialog-Funktion staged Buchungen als pendingDayChanges — diese durchlaufen commitDayChanges() und erhalten damit automatisch die neue Chat-Karte
  • Deposit-Daten aus Escrow: Aktuell wird depositAmount aus ProfileStore gelesen (Teacher-Einstellung). Präziser wäre der Wert aus dem tatsächlichen Escrow-Objekt nach writeBlockEscrowHold()
  • Karten für vergangene plain-text Buchungen: Alte Nachrichten (vor diesem Update) bleiben als plain text — kein Migrationsbedarf, da rückwärtskompatibel
