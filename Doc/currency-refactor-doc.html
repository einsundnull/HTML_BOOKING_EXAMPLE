1 Ziel & Entscheidungen

Ziel dieses Refactors ist es, alle hardcodierten €-Symbole und fmtPrice()-Aufrufe ohne Currency-Code durch korrekte, nutzerabhängige Währungsformatierung zu ersetzen. Der CurrencyService mit Mockup-Rates war bereits vollständig implementiert — er wurde nur nicht konsequent genutzt.

Architekturentscheidungen (vor Implementierung festgelegt)
  • Wallet-Währung: Admin-Seiten immer EUR, alle anderen Seiten: displayCurrency des eingeloggten Users
  • financial.js & admin.js: Phase 1 übersprungen — nur student/teacher-facing fixes
  • Reihenfolge: profile → student → teacher → ui (shared helpers zuerst, Consumer danach)
  • Konversionsbasis: Alle internen Preise sind in EUR gespeichert. Display-Konversion läuft immer via convertSync(amount, 'EUR', displayCurrency)
  • Fallback: Bei kaltem Cache oder fehlendem CurrencyService → immer EUR, kein Crash

2 Architektur

Der Kern des Refactors ist ein neuer globaler Helper _fmtForUser() in ui.js. Statt in jeder Datei denselben CurrencyService-Lookup zu duplizieren (wie es in den bereits korrekten Stellen in student.js der Fall war), wird die Logik zentralisiert:

/* Vorher: duplizierter Lookup in jedem Aufrufer */
var sCur = CurrencyService.getUserCurrency(currentUser.uid);
if (sCur === 'EUR') return fmtPrice(amount, 'EUR');
var conv = CurrencyService.convertSync(amount, 'EUR', sCur);
if (conv === null) return fmtPrice(amount, 'EUR');
return CurrencyService.format(conv, sCur);

/* Nachher: ein zentraler Aufruf */
return _fmtForUser(amount, currentUser.uid);

Die Konversionskette bleibt identisch — nur der Ort der Logik ändert sich. CurrencyService.warmUp() läuft bereits beim Seitenstart via DOMContentLoaded, sodass der Rate-Cache bei der ersten User-Interaktion garantiert warm ist.

4 Neuer Helper: _fmtForUser()

Definiert in ui.js, global verfügbar über window._fmtForUser.

/**
 * _fmtForUser(amountEUR, uid)
 *
 * Formatiert einen EUR-Betrag in der displayCurrency des Users.
 * Liest displayCurrency über CurrencyService.getUserCurrency(uid),
 * der wiederum ProfileStore.get(uid).displayCurrency liest.
 *
 * Fallback-Kette:
 *   1. CurrencyService nicht geladen  → fmtPrice(amount, 'EUR')
 *   2. uid fehlt / kein Profil        → fmtPrice(amount, 'EUR')
 *   3. displayCurrency === 'EUR'      → fmtPrice(amount, 'EUR')
 *   4. convertSync gibt null zurück   → fmtPrice(amount, 'EUR')  (Cache kalt)
 *   5. Alles OK                       → CurrencyService.format(converted, cur)
 */
function _fmtForUser(amountEUR, uid) {
  if (typeof CurrencyService === 'undefined') return fmtPrice(amountEUR, 'EUR');
  var cur = (uid && typeof ProfileStore !== 'undefined')
    ? CurrencyService.getUserCurrency(uid)
    : 'EUR';
  if (!cur || cur === 'EUR') return fmtPrice(amountEUR, 'EUR');
  var converted = CurrencyService.convertSync(amountEUR, 'EUR', cur);
  if (converted === null) return fmtPrice(amountEUR, 'EUR');
  return CurrencyService.format(converted, cur);
}
window._fmtForUser = _fmtForUser;

Aufruf-Konventionen

/* Student-Kontext */
_fmtForUser(amount, currentUser ? currentUser.uid : null)

/* Teacher-Kontext */
_fmtForUser(amount, currentUser ? currentUser.uid : null)

/* Shared UI-Helper ohne direkten currentUser-Zugriff */
_fmtForUser(amount, _getActorUid())

5 Neuer Helper: _getActorUid()

Shared UI-Funktionen wie _buildConfirmFilterRow() haben keinen direkten Zugriff auf currentUser. Dieser Helper löst das auf:

function _getActorUid() {
  if (typeof currentUser !== 'undefined' && currentUser) return currentUser.uid;
  if (typeof Auth !== 'undefined' && Auth.current) {
    var u = Auth.current();
    return u ? u.uid : null;
  }
  return null;
}
window._getActorUid = _getActorUid;

11 Offene Punkte & Hinweise

1. Doppelter Lokaler Helper in student.js

Die Funktionen _fmtInStudentCur() (in _renderSaveSummary) und _fmtD() (in _confirmBookings) sind lokale Inline-Varianten von _fmtForUser() — entstanden bevor der zentrale Helper existierte. Sie können in einem separaten Cleanup-Schritt durch _fmtForUser(amount, currentUser.uid) ersetzt werden. Funktional korrekt, aber redundant.

2. wallet-core.js — Signaturänderung mit Risiko

WalletCore.formatAmount() wird an mehreren Stellen aufgerufen (wallet.js, wallet-panel.js, student.js, teacher.js). Die Erweiterung um einen optionalen currencyCode-Parameter ist rückwärtskompatibel, aber alle Aufrufer müssen geprüft werden.

3. Currency-Cache beim Seitenstart

CurrencyService.warmUp() ist async (setTimeout 0). Bei sehr schnellen Interaktionen direkt nach Seitenload kann convertSync() noch null zurückgeben und EUR-Fallback zeigen. Dies ist by design und kein Bug — nach dem ersten Render ist der Cache warm.

4. teacher.js — walletBalance ist null

In Teacher-Confirm-Dialogen wird walletBalance: null übergeben (Teacher sieht kein eigenes Wallet im Confirm). Daher zeigt _buildConfirmDetailBody keine "Guthaben danach"-Zeile — korrekt so.
