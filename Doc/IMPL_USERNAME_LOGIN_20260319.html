Username — Login per E-Mail oder Benutzername
Vollständige 4-Layer Implementierung: Store → Adapter → AppService → Consumer. Kein direkter Store-Zugriff im Consumer-Code.
📅 19. März 2026 7 Dateien · APP_VERSION 2026-03-19-53
1
4-Layer Architektur
Consumer
landing.js · profile.js · admin.js · auth.js
Ruft nur AppService.* und Auth.* auf. Kein direkter Store.*-Zugriff. Keine localStorage-Calls.
↓
Service
app-service.js
Thin wrapper. Delegiert via _call(methodName, args) an den aktiven Adapter. Exponiert die öffentliche API für Consumer.
↓
Adapter
adapter-localstorage.js
Implementiert die Logik: Slugify, Uniqueness-Check, Fallback-Nummerierung. Einzige Stelle die Store.* direkt aufruft.
↓
Store
store.js
Datenhaltung. Users.create(), Users.update(), Users.byUsername(). Kein Business-Logic.
auth.js ist Infrastructure (kein Consumer-Code) und darf Store.* für current() und login() nutzen — diese brauchen keine Async-Adapter-Indirektion. loginByEmailOrUsername() hingegen nutzt AppService, da es eine Suchabfrage über alle User macht.

Layer 1 — store.js: Neue Methoden
MethodeSignaturBeschreibung
Users.byUsername(u)(string) → user|nullCase-insensitive Lookup. Gibt erstes Match oder null.
Users.create(opts)({uid,name,role,username,...}) → userNeu: username-Feld, Uniqueness-Check beim Erstellen.
Users.update(uid, patch)(string, {username?,...}) → voidNeu: username in patch erlaubt + Uniqueness-Check.
User-Objekt-Schema (neu): { uid, name, username, role, email, password, discipline }. username ist immer lowercase, ohne @-Prefix, 3–30 Zeichen, nur [a-z0-9._]. Bestehende User ohne Username haben username: ''.

Layer 2 — adapter-localstorage.js: Neue Funktionen
FunktionSignaturLogik
isUsernameAvailable(username, cb(err, bool))Validiert Format (3–30, [a-z0-9._]), prüft Store.Users.byUsername().
generateUsername(name, cb(err, string))Slugify: „Max Müller" → max.muller. Falls besetzt: max.muller2, max.muller3 … bis 999.
getUserByEmailOrUsername(query, cb(err, user|null))Enthält @domain → E-Mail-Suche. Sonst: Username-Suche (führendes @ wird entfernt). Gibt null (kein Error) wenn nicht gefunden.
_slugify(name)(string) → stringPrivat. Lowercase, Leerzeichen→Punkt, Non-ASCII entfernt, max 28 Zeichen.
E-Mail vs. Username Erkennung
/* Regex: enthält @ gefolgt von Domain → E-Mail */ var isEmail = /^[^@]+@[^@]+\.[^@]+$/.test(q); /* Beispiele: */ // "max@gmail.com" → isEmail = true → E-Mail-Suche // "@max.mueller" → isEmail = false → Username (@ entfernt) // "max.mueller" → isEmail = false → Username direkt

Layer 4a — auth.js: loginByEmailOrUsername
auth.js
Auth.loginByEmailOrUsername(query, password, onSuccess, onError) /* query = E-Mail oder @username oder username (ohne @) password = Klartext onSuccess(user) — Redirect wird intern ausgeführt onError(msg) — Consumer zeigt Fehlermeldung */ /* Flow: */ AppService.getUserByEmailOrUsername(query, function(err, user) { // → prüft password → Auth.login(user.uid) → Redirect });
Async-Callback statt throw. loginByEmail() ist synchron (throw). loginByEmailOrUsername() ist async wegen AppService-Adapter. Consumer-Code (landing.js) nutzt onSuccess/onError-Callbacks — kein try/catch nötig.
