Feature 2 — Verschiebungs-Audit-Log
Neuer Storage-Key: app_move_log — Array von MoveLogEntry-Objekten. Wird bei jeder Verschiebung automatisch befüllt. Keine manuelle Aktion nötig.

Vollständige Aufrufkette bei Verschiebung
ui.js → student.js / teacher.js → adapter → chat.js → email.js
1
ui.js — openMoveBlockDialogShared()
Dialog öffnet mit Datum/Zeit + Grund-Sektion
Grund-Dropdown wird mit MOVE_REASONS gefüllt, gefiltert nach opts.actorRole.
2
ui.js — confirmBtn.click
Pflichtfeld-Validierung
Kein Grund → .input-error auf Select, Fehlermeldung, Abbruch. Sonst: opts._lastMoveOpts = { reason, reasonLabel, note } gesetzt, opts.onConfirm(pending, opts) aufgerufen.
3
student.js / teacher.js — onConfirm()
pendingMoveOpts[partyId] = opts._lastMoveOpts
Move-Opts werden nach teacherId (Student) oder studentId (Lehrer) zwischengespeichert, für den späteren writeMoveRecord-Aufruf nach Save.
4
student.js / teacher.js — onAllDone() / confirmAndSave()
AppService.writeMoveRecord(oldSlot, newSlot, uid, role, cb, moveOpts)
Adapter schreibt: TX (move) für Initiator + Gegenseite, Store.Slots.markMoved(), Store.MoveLog.add() mit reason+note+movedBy.
5
student.js / teacher.js — im writeMoveRecord-Callback
ChatStore.sendMoveNotification(senderId, receiverId, snap)
Chat-Nachricht Typ booking_moved an die Gegenseite mit altem/neuem Termin, Grund, Freitext.
6
student.js / teacher.js
EmailService.onBookingMoved(receiverUid, data)
E-Mail an die Gegenseite. Bereits implementierte API wird jetzt erstmals aufgerufen.
