Überblick — Zwei Bestätigungspfade
PfadEntry PointAusgelöst durchKernfunktionen
Block-Pfad
(mehrere Slots)		 AppService.confirmBlock(slots, cb) Student oder Teacher klickt „Bestätigen" auf einem gebuchten Block confirmSlot × N → getEscrowBySlot × N → confirmLessonSilent × N → writeBlockLessonConfirmed × 1
Einzelslot-Pfad
(ein Slot)		 AppService.confirmSlot(slotId, cb) Einzelner Slot schnell bestätigen confirmSlot (adapter) → getEscrowBySlot → confirmLesson
Wichtige Designentscheidung — Block-Pfad: Wallet-Updates und Escrow-Zustandsänderungen passieren pro Slot in confirmLessonSilent (kein TX). Danach schreibt writeBlockLessonConfirmed genau einen aggregierten TX-Eintrag für den gesamten Block — nicht N einzelne. Das hält die Transaktionshistorie lesbar.
Block-Pfad — vollständige Aufrufkette in Reihenfolge
app-service.js → adapter-localstorage.js
1
app-service.js
confirmBlock(slots, cb)
Einstiegspunkt. Empfängt slots[] — können Stubs {slotId} sein. Schritt 0: Reichert Stubs mit vollständigen Slot-Objekten aus getAllSlotsSync() an → fullSlots[] mit .price, .teacherId, .studentId, .date, .time. Startet dann Step 1.
kein Wallet kein TX kein Escrow
2
adapter-localstorage.js — via _call('confirmSlot') × N
confirmSlot(slotId) — pro Slot
Setzt slot.status = 'confirmed' im Store. Keine Wallet-Änderungen. Alle N Aufrufe laufen, dann afterConfirm().
slot.status = 'confirmed' kein Wallet kein TX
3
app-service.js — afterConfirm()
getEscrowBySlot(slotId) × N
Lädt den Escrow für jeden Slot. Nur Escrows mit depositStatus === 'held' werden in confirmedEscrows[] gesammelt. Slots ohne Escrow (cash_on_site) werden ignoriert. Wenn alle fertig: afterEscrows().
nur Lesen confirmedEscrows[] befüllt
4
adapter-localstorage.js — via _call('confirmLessonSilent') × N
confirmLessonSilent(escrowId) — pro Escrow
Kernfunktion für Wallet-Updates. Pro Escrow:
  1. Berechnet remainderAmt = fullAmount − depositAmount
  2. Zieht remainderAmt vom Schüler-Wallet ab (der Deposit wurde bereits bei Buchung abgezogen)
  3. Schreibt fullAmount dem Lehrer-Wallet gut (= Deposit + Restbetrag)
  4. Setzt escrow.depositStatus = 'released'
  5. Setzt escrow.fullPaymentStatus = 'verified'
  6. Setzt escrow.fullPaidAt und escrow.releasedAt
Schreibt keinen TX — das macht writeBlockLessonConfirmed aggregiert.
student.balance -= remainder teacher.balance += fullAmount status = released fullPaidAt = now kein TX
5
adapter-localstorage.js — via _call('writeBlockLessonConfirmed') × 1
writeBlockLessonConfirmed(fullSlots, confirmedEscrows, cb)
Schreibt die TX-Aufzeichnungen. Wallet-Salden wurden bereits in Step 4 aktualisiert. Diese Funktion liest nur die aktuellen Salden und erstellt zwei TX-Einträge:
  • Lehrer-TX (escrow_release): amount = totalFull (Summe aller Slot-Preise)
  • Schüler-TX (lesson_confirmed): amount = −totalRemainder
totalFull kommt aus fullSlots[].price (deshalb müssen fullSlots verwendet werden, nicht Stubs).
totalRemainder = totalFull − totalDep (totalDep aus confirmedEscrows[].depositAmount).
kein Wallet (bereits done) TX: escrow_release (Lehrer) TX: lesson_confirmed (Schüler)
6
student.js / teacher.js
cb() → UI-Update
Toast-Meldung. Wallet-Panel refresh (WalletPanel.refresh(uid)). Buchungsliste und Statistiken neu laden.
UI only

Escrow-Zustandsmaschine
depositStatusfullPaymentStatusBedeutungGesetzt durch
unpaidunpaid Escrow erstellt, noch kein Deposit createEscrow()
heldunpaid Deposit vom Schüler gezahlt, wartet auf Bestätigung payDeposit() / payDepositSilent()
releasedverified Stunde bestätigt, Vollzahlung abgeschlossen confirmLessonSilent() / confirmLesson()
refund_requestedunpaid Stornierung angefragt requestDepositRefund()
refundedunpaid Deposit zurückgezahlt releaseDeposit()
forfeitedunpaid Deposit verfallen (Stornierung zu spät) forfeitDeposit()
fullPaymentStatus-Felder: unpaid → paid → verified. Der paid-Schritt existiert im Schema aber wird nicht aktiv gesetzt — der Übergang von unpaid zu verified erfolgt direkt in confirmLessonSilent / confirmLesson. fullPaidAt wird bei Bestätigung gesetzt.
TX-Einträge — was wo geschrieben wird
ZeitpunktFunktionuidtypeamountdescription
BuchungwriteBlockBookingRecord Teacherbooking0 „Neue Buchung — N Slots am..."
BuchungwriteBlockEscrowHold Studentescrow_hold−totalDeposit „Deposit — N Slots am..."
BestätigungwriteBlockLessonConfirmed Teacherescrow_release +totalFull „Zahlung freigegeben — N Slots am..."
BestätigungwriteBlockLessonConfirmed Studentlesson_confirmed −totalRemainder „Stunde bestätigt — N Slots am..."
TX amount für lesson_confirmed: Zeigt −totalRemainder (negativer Betrag = Abbuchung). Wenn kein Restbetrag (z.B. Deposit = Vollpreis), dann amount = 0. Der Saldo danach (balance) im TX-Eintrag spiegelt das tatsächliche Guthaben nach allen Operationen wider und wird aus dem bereits aktualisierten Wallet gelesen.
