Booking Notification — Chat Service Message
Strukturierte, gebündelte Buchungsbenachrichtigung im Chat. Ersetzt die alten Freitext-Nachrichten durch eine klickbare Service-Card mit Preis, Zeitblöcken und Wallet-Overlay.
📅 19. März 2026 4 Dateien geändert chat.js · student.js · teacher.js · chat.css Typ: booking_notification
Motivation & Problem
Vorher: Buchungen wurden als einfache Textnachrichten gesendet (📅 Neue Buchung: Do., 19. März 2026, 14:00–16:30 (5 Slots)). Pro Datum eine eigene Nachricht → mehrere Buchungen an verschiedenen Tagen = mehrere separate Nachrichten. Kein Preis, kein Wallet-Link, nicht klickbar.
❌ Alt

Zwei separate Textnachrichten für zwei Buchungsdaten.
Kein Preis, kein Detail-Link.

📅 Neue Buchung: Do., 19. März 2026, 14:00–16:30 (5 Slots)
📅 Neue Buchung: Do., 26. März 2026, 14:00–16:30 (5 Slots)
📅 Neue Buchung: Do., 19. März 2026, 17:30–18:30 (2 Slots)
✅ Neu

Eine strukturierte Service-Card pro Speichervorgang. Alle Daten gebündelt. Preis sichtbar. Klickbar → Overlay → Wallet.

Trigger-Flow — beide Richtungen
Die Nachricht wird in beide Richtungen gesendet: Student bucht → Teacher empfängt. Teacher bucht für Student → Student empfängt. Jeweils der Buchende ist senderId.
Trigger A — Student bucht
1
student.js — saveDayChanges()
User drückt Speichern. pendingDayChanges enthält die gebuchten Slots.
2
bookSlotWithEscrowSilent() async
Slots werden in localStorage gebucht. Preis wird via getTeacherPriceSync() synchron gelesen vor dem async Teil.
3
notifGroups aufbauen
Alle gebuchten Slots werden nach teacherId → date → slots[] gruppiert. slotIds und slotTimes werden je Slot gesammelt.
4
ChatStore.sendBookingNotification()
sender = student.uid, receiver = teacher.uid, actorRole = 'student'. Alle Daten im Snapshot.
Trigger B — Teacher bucht für Student
1
teacher.js — saveBookingGrid()
pendingBookings enthält die Zuordnungen Student → Slot.
2
bookSlotWithEscrowSilent() async
Identisch wie Student-Flow. Preis aus eigenem Profil per getTeacherPriceSync(currentUser.uid).
3
notifByStu aufbauen
Erst bookGroups nach stuId_date, dann re-gruppiert nach stuId only → alle Daten in einem blocks[].
4
ChatStore.sendBookingNotification()
sender = teacher.uid, receiver = student.uid, actorRole = 'teacher'.
Preis & Währungskonvertierung
Wichtig: Beträge im Snapshot sind immer in EUR gespeichert (Systemwährung). Die Anzeige im Chat wird in der displayCurrency des eingeloggten Users gerendert — via _fmtForUser(amountEUR, uid).
SchichtWertFunktion
Snapshot (gespeichert) pricePerSlot: 12.50, totalAmount: 125.00 Immer EUR. AppService.getTeacherPriceSync() → liest Profil-Preis und konvertiert zu EUR via CurrencyService.convertSync().
Card-Rendering _fmtForUser(amount, actorUid) Konvertiert EUR → displayCurrency des Viewers. Fallback auf EUR wenn Cache kalt.
Overlay-Rendering lokale fmt() Funktion Gleiche Logik: _fmtForUser → fmtPrice → Hardcode-Fallback. Dreistufig.

"Im Wallet anzeigen" — Navigationslogik
1
Overlay schließen
Immer zuerst. Overlay wird mit Animation geschlossen.
2
WalletPanel verfügbar? (same-page Panel)
Check: typeof WalletPanel !== 'undefined' && typeof WalletPanel.scrollToTx === 'function'.
3a
✅ Ja → In-Page-Navigation
document.getElementById('nav-wallet').click() → öffnet Wallet-Tab.
Nach 350ms: WalletPanel.scrollToTx(null, firstSlotId, actorUid) → scrollt zur ersten Buchungs-TX.
3b
❌ Nein → Seitennavigation
Navigiert zu wallet.html?uid=...&highlightSlot=firstSlotId.
Der highlightSlot-Parameter kann von wallet.js ausgelesen werden, um die relevante TX direkt zu öffnen.
Architektur-Anmerkungen
Keine Store.*-Direktzugriffe. Alle Datenzugriffe laufen über AppService.* und ProfileStore.* — konform mit Guideline_Architecture.
Keine inline styles. Alle Styles in chat.css. Keine style="..." Attribute im gerenderten HTML — konform mit STYLE-GUIDE.
Kein ES6+. Kein let/const, keine Arrow-Functions, keine Template-Literals, kein ?./?? — konform mit JS-Guideline.
Preis-Timing: getTeacherPriceSync() wird synchron vor dem async bookSlotWithEscrowSilent() aufgerufen. Das ist korrekt — zu diesem Zeitpunkt ist der Preis im Lehrerprofil verfügbar. Der Preis, der tatsächlich auf dem Slot gespeichert wird, kann minimal abweichen falls ein zweiter bookSlot-Aufruf gleichzeitig läuft, aber für die Benachrichtigungs-Anzeige ist das akzeptabel. Der echte TX-Betrag ist immer in der Wallet sichtbar.
Rückwärtskompatibilität: Alte text-Nachrichten mit dem 📅 Neue Buchung:-Format bleiben unverändert lesbar — der _buildTextBubble-Renderer ist nicht geändert. Nur neue Buchungen erzeugen booking_notification-Messages.
