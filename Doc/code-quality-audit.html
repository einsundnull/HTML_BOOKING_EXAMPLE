R1 Race Condition Synchrone Renders vor asyncem Schreiben — mehrere Ops
commitDayChanges() / saveBookingChanges() feuern n parallele bookSlotWithEscrow- und cancelSlotWithPolicy-Calls in einer Schleife. pendingDayChanges = {} und die UI-Renders folgen synchron — also bevor irgendein Callback zurückgekommen ist. Bei 3 gebuchten Slots z.B. feuern 3 unabhängige Callbacks, die alle nochmals renderCalendar() aufrufen — 6 Renders total, keine Reihenfolgengarantie, kein Barrier/Counter. 		student.js ~236
teacher.js ~1044		 Flackernde UI bei mehreren gleichzeitigen Buchungen. Letzter Callback "gewinnt". Hoch Promise.all-Äquivalent: Counter-Barrier (var done=0; if(++done===total)) — erst wenn alle Callbacks zurück, ein einziger Re-render + Wallet-Refresh.
R2 Race Condition Double-Save möglich — kein Busy-Guard
Weder confirmAndCommit() noch saveBookingChanges() setzen einen _saving-Flag oder deaktivieren den Save-Button für die Dauer der asynchronen Schleife. Schnelles Doppelklicken auf "Speichern" schickt dieselben Slot-IDs zweimal in die Adapter-Schicht. 		student.js ~165
teacher.js ~1044		 Doppelte Escrow-Anlage, doppelter Deposit-Abzug möglich. Hoch btn.disabled = true sofort beim Klick, btn.disabled = false erst im letzten Callback (Barrier).
R3 Race Condition Navbar-Wallet in teacher.js Cancel-Pfad doppelt aktualisiert
Im Cancel-Callback von saveBookingChanges() wird AppService.getWallet aufgerufen und das Badge inline gesetzt (Zeile ~1087). Direkt danach ruft das Ende von saveBookingChanges() WalletPanel.refresh() auf, das ebenfalls AppService.getWallet aufruft und via _onBalanceUpdate dasselbe Badge setzt. Zwei parallele async Calls auf dasselbe DOM-Element — letzter gewinnt. 		teacher.js ~1087 + ~1132 Kein falsches Verhalten, aber zwei unnötige getWallet-Requests pro Cancel-Op. Mittel Inline-Navbar-Refresh im Cancel-Callback entfernen — WalletPanel.refresh() am Ende deckt beides ab.
