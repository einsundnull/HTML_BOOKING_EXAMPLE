Username-Regeln
RegelWert/Format
Erlaubte Zeichen[a-z0-9._] — Kleinbuchstaben, Zahlen, Punkt, Unterstrich
Länge3–30 Zeichen
Gross-/KleinschreibungImmer lowercase gespeichert. Eingabe case-insensitive.
@-PrefixNur im UI angezeigt (@matthias.h). Im Objekt ohne @: matthias.h.
Auto-GenerierungAus name: Leerzeichen → Punkt, Non-ASCII entfernt. Kollision → Suffix 2, 3, …
EindeutigkeitSystemweit unique. Check in Store.Users.byUsername() und AppService.isUsernameAvailable().
ÄnderungErlaubt via AppService.updateUser(uid, {username}, cb). Uniqueness wird geprüft.
Bestehende Userusername: '' — kein Username. Login nur per E-Mail oder UID.
✅ Gültige Benutzernamen
matthias.h · max_mueller · ski.pro2 · anna123
❌ Ungültige Benutzernamen
ab (zu kurz) · Matthias H (Leerzeichen, Großbuchstabe) · max@ski (@ im Handle) · test! (Sonderzeichen)

Auth-Regeln
MethodeWann verwendenSynchron?
Auth.current()Auf geschützten Seiten: Wer ist eingeloggt? Liest ?uid= aus URL.✅ Sync
Auth.require(role)Guard am Seitenanfang. Redirectet wenn falsche Rolle oder nicht eingeloggt.✅ Sync
Auth.login(uid)Interner Redirect nach bekannter uid. Nur von Auth-Methoden aufrufen.✅ Sync
Auth.loginByEmailOrUsername()Sign-In Formular auf Landing-Page. E-Mail oder @Handle.❌ Async (Callback)
Auth.loginByEmail()Veraltet für neue Features. Nur noch intern/legacy. Bevorzuge loginByEmailOrUsername.✅ Sync (throw)
Auth.logout()Abmelden. Redirectet zu landing.html.✅ Sync
Auth.loginByGoogle(uid)Google-Mock. Ruft Auth.login(uid) auf.✅ Sync
✅ Korrekt — Async Error-Handling bei loginByEmailOrUsername
Auth.loginByEmailOrUsername(query, pw,
  function() { /* Redirect erfolgt intern */ },
  function(msg) { self._showSigninError(msg); }
);
❌ Falsch — try/catch mit async Auth-Methode
try { Auth.loginByEmailOrUsername(…); } catch(e) { … } // funktioniert NICHT — async!

Login-Erkennung (E-Mail vs. Username): Enthält die Eingabe @ gefolgt von einer Domain (Regex /^[^@]+@[^@]+\.[^@]+$/) → E-Mail-Suche. Alles andere → Username-Suche. Führendes @ im Handle wird automatisch entfernt.

5
Verbotene Muster
❌ Store.* in Consumer-Code
Store.Users.all(), Store.Users.byUid(), Store.Users.byUsername() etc. sind ausschließlich dem Adapter erlaubt.
❌ ProfileStore.* für Auth-Lookup
ProfileStore nur für Anzeige-Namen und Profil-Daten. Niemals für Login-Identifikatoren.
❌ uid als Login-Handle verwenden
Der interne uid (t1, u_t_123) darf nicht als Login-Identifier im UI exponiert werden. Nur username und email sind Login-Identifier.
❌ Realname als eindeutigen Identifier behandeln
name ist NICHT eindeutig. Zwei User können „Max Müller" heißen. Für Uniqueness immer username oder email verwenden.
❌ Username mit @ im Objekt speichern
Im User-Objekt wird username ohne @-Prefix gespeichert. Das @ ist nur ein UI-Indikator. Beim Speichern immer .replace(/^@/, '') aufrufen.
❌ Synchronen try/catch bei async Auth-Methoden
loginByEmailOrUsername() ist async. Fehler kommen über den onError-Callback, nicht über exceptions.
