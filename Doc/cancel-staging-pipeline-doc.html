Buchungs-Pipeline

Vollständige Dokumentation · BookingSystem · 18.03.2026

1 — Überblick: Staged-Commit-Prinzip

Das System verwendet ein Staged-Commit-Modell — alle Änderungen (Buchen, Stornieren) werden zuerst lokal in einer pendingMap vorgemerkt (gestaged). Erst beim Klick auf Done / Speichern werden sie tatsächlich ausgeführt. Solange etwas staged ist, kann es per Undo rückgängig gemacht werden.

Kernprinzip: Kein API-Aufruf, keine Datenbankschreibung, keine E-Mail — bis der Nutzer explizit bestätigt. Der Policy-Dialog erscheint erst am Ende, nicht beim Klick auf Cancel.

↓

↓

Phase B — Commit (Klick auf „Done")

1
Schüler klickt Done → confirmAndCommit()
↓
2
Prüfung: gibt es staged Cancels? (pendingDayChanges.filter(action==='cancel'))
↓

Zeigt: Verbleibende Zeit · Tier (Volle/Teil/Forfeit) · Betrag
Buttons: Nicht stornieren / Jetzt stornieren
↓
4a
„Nicht stornieren": Cancel aus pendingDayChanges entfernt → Slot wieder normal → nächster Cancel-Dialog
4b
„Jetzt stornieren": Cancel bleibt in pending → nächster Cancel-Dialog
↓
5
Alle Cancel-Dialoge abgearbeitet → Booking-Dialog (falls es auch staged Buchungen gibt)
↓
6
commitDayChanges() → führt alle verbleibenden pending-Änderungen aus:
• book → AppService.bookSlotWithEscrowSilent(..., 'student')
• cancel → AppService.cancelSlotWithPolicy(sid, 'student', cb)
