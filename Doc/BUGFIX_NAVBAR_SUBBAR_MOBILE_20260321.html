BUGFIX · UI · Landing Page · BookingSystem
Navbar Sub-Bar — Mobile unsichtbar
Auf landing.html (mobil, ≤768px) war der Sub-Bar mit den Währungs- und Sprachschaltern vollständig unsichtbar. Das Element war korrekt im DOM vorhanden, wurde aber von der Hero-Sektion überlagert, weil <main> keinen padding-top hatte, der die Höhe von navbar (52px) + sub-bar (36px = 88px gesamt) berücksichtigt.
🟡 Visual · Mittel 📅 21. März 2026 3 Dateien geändert · landing.css · landing.js · app-config.js ✅ Behoben · APP_VERSION 2026-03-21-30
1

Root Cause

Kern des Problems: Die Navbar ist position: fixed und nimmt keinen Platz im Dokumentfluss ein. Der Sub-Bar ist position: static und befindet sich im Dokumentfluss — direkt nach der Navbar, also an y=52px. <main> auf landing.html hatte kein padding-top, weshalb die Hero-Sektion ab y=0 begann und sowohl Navbar als auch Sub-Bar überlagerte. Der Sub-Bar war im DOM vorhanden, aber vollständig hinter der Hero-Sektion versteckt.
Warum Desktop nicht betroffen war: Auf Desktop (≥769px) ist .navbar-subbar { display: none } — er wird gar nicht gerendert. Die Währungs- und Sprach-Pills sitzen dort in .navbar-selects-desktop innerhalb der Hauptnavbar-Zeile. Das fehlende padding-top auf <main> war auf Desktop zwar ebenfalls vorhanden, aber die 50px-Überlappung fiel im Hero-Bereich (der dunkel und höhengroßzügig ist) optisch nicht auf. Auf Mobile war der 88px-Offset entscheidend.

Wichtig — kein gleichzeitiges Rendering: Sub-Bar und .navbar-selects-desktop werden nie gleichzeitig gezeigt. Die CSS-Regeln @media (max-width: 768px) { .navbar-selects-desktop { display: none } } und @media (min-width: 769px) { .navbar-subbar { display: none } } stellen das sicher. Kein JS-Eingriff nötig.

Implementierter Fix

3 Dateien · landing.css · landing.js · app-config.js
Fix-Strategie: Reines CSS-Fix — kein JS, kein Inline-Style. main erhält in landing.css ein padding-top: 52px (Desktop, nur Navbar) und per Media Query padding-top: 88px (Mobile, Navbar 52px + Sub-Bar 36px). Zusätzlich wird in landing.js die navH-Berechnung für den Scroll-Jump-Button korrigiert, um die Sub-Bar-Höhe einzuschließen.

4

Design-Entscheidung — Sub-Bar bleibt position: static

Warum kein position: fixed? Der Sub-Bar ist ein Landing-Page-only-Element für unauthentifizierte Besucher. Er scrollt intentional aus dem Sichtbereich wenn der User nach unten scrollt — genau wie alle anderen Landing-Page-Sektionen. Die fixe Navbar (mit dem Smart-Scroll-Behavior) ist das einzige persistente Chrome-Element. Einen zweiten fixierten Balken würde wertvolle vertikale Fläche auf Mobile dauerhaft blockieren. position: static ist das korrekte Verhalten.
Konsequenz: Das CSS-padding-top auf main ist der einzig korrekte Fix. Der Sub-Bar nimmt seinen 36px Platz im Dokumentfluss ein — main muss nur wissen, dass er erst nach Navbar + Sub-Bar beginnt.
