Recurring Booking — Price & Deposit Display
Beide recurring-Buchungsdialoge zeigen jetzt vollständige Preis- und Depositinformationen: Preis pro Wochenzeile, aufgeschlüsselter Kosten-Block (Deposit jetzt / Restbetrag / Bar vor Ort), Guthabenprüfung und Gesamtpreis im Bestätigungs-Button. Währungsumrechnung über _fmtForUser().

— (Button versteckt) —
canProceed-Logik: Der Bestätigungs-Button wird nur angezeigt wenn availCount > 0 — unabhängig vom Guthaben. Das bedeutet: ein Schüler kann auch bei zu niedrigem Guthaben technisch fortfahren. Die Warnung ist informativ, kein harter Block — konsistent mit dem Verhalten des regulären Buchungsmodals.
Ausführungsfluss — Dialog 1 (buildStudentBookingDialog)
recurCheckBtn → async → render
1
Verfügbarkeit prüfen geklickt
recurCheckBtn.addEventListener('click', ...) feuert. pricePerSlot wird via AppService.getTeacherPriceSync(teacherId) gelesen — synchron, kein Callback nötig.
2
Wochenverfügbarkeit berechnen (sync)
Schleife W=1…N: Datum berechnen, Slots materialisieren, isFreeSlot()-Check. Ergebnis in recurData[]. availCount und totalCost berechnen.
3
Deposit-Info holen (async)
Wenn pricePerSlot > 0: AppService.calcDepositInfo(teacherId, pricePerSlot, cb). Bei Fehler oder pricePerSlot === 0: _renderRecurResult(null) direkt aufrufen (kein Deposit, kostenlos).
4
_renderRecurResult(dep) — im Callback
Wallet sync lesen. payMode, reqDep, depAmt, depTotal aus dep extrahieren. fmt()-Funktion bauen (_fmtForUser oder Fallback).
5
HTML-Rendering nach Zahlungsmodus
Branch: pricePerSlot===0 → kostenlos. cash_on_site → Bar-Zeile. !reqDep → Vollpreis + Guthabencheck. reqDep → Deposit-Aufschlüsselung (jetzt / pro Slot / Restbetrag / Guthaben).
6
Wochenzeilen rendern mit Preis-Badge
Pro recurData[r]: Icon, Datum+Uhrzeit, dann Preis-Badge: recur-row-price (Deposit oder Vollpreis), recur-row-price--cash (Bar vor Ort, kursiv), oder leer bei nicht-verfügbaren Zeilen.
7
Button-Label bauen
„N Termine buchen" + Suffix: „ — Bar vor Ort" / „ — Deposit: €X" / „ — €Y" / kein Suffix bei kostenlos. Button-Visibility hängt nur von availCount > 0 ab.
Schlüsselvariablen & Berechnungen
VariableQuelleBedeutung
pricePerSlot AppService.getTeacherPriceSync(teacherId) Preis pro Einzelslot in EUR (synchron). 0 = kostenlos.
totalCost availCount × slotCount × pricePerSlot Gesamtkosten aller verfügbaren Wochen. In Dialog 2: availCount × pricePerSlot (slotCount=1).
dep.paymentMode AppService.calcDepositInfo() callback 'instant' | 'cash_on_site'. Bei cash_on_site: kein Guthaben nötig.
dep.requiresDeposit AppService.calcDepositInfo() callback true = Deposit erforderlich. false = Vollpreis sofort.
dep.depositAmount AppService.calcDepositInfo() callback Depositbetrag pro Slot (bereits anteilig berechnet). Kann fixed oder percent sein.
dep.depositType AppService.calcDepositInfo() callback 'fixed' | 'percent'. Steuert Depositlabel-Text.
dep.depositPercent AppService.calcDepositInfo() callback Nur bei depositType='percent'. Z.B. 20 → „20% pro Slot".
depTotal depAmt × availCount (gerundet auf Cent) Gesamter Depositbetrag für alle verfügbaren Wochen.
restTotal max(0, totalCost − depTotal) Restbetrag fällig bei Bestätigung der Stunden.
fmt(amount) Closure: _fmtForUser(a, stuId) oder Fallback Konvertiert EUR-Betrag in die Anzeigewährung des Schülers.
calcDepositInfo-Aufruf: Der Aufruf erfolgt mit pricePerSlot (Einzelslotpreis), nicht mit totalCost. Der Adapter berechnet intern den Depositbetrag basierend auf dem Einzelpreis (bei percent-Deposit: pricePerSlot × percent / 100). Die Multiplikation mit availCount erfolgt danach im Frontend.

Fehlerbehandlung & Fallbacks
SzenarioVerhalten
calcDepositInfo gibt Fehler zurück Dialog 1: _renderRecurResult(null) → rendert ohne Deposit-Info (wie kostenlos). Dialog 2: dep wird via (!errDep && dep)-Guards abgesichert → alle Deposit-Flags fallen auf Fallback-Werte.
pricePerSlot === 0 Dialog 1: calcDepositInfo wird gar nicht aufgerufen, direkt _renderRecurResult(null). Dialog 2: calcDepositInfo wird immer aufgerufen (keine Preis-Prüfung im outer scope). Beide rendern „Kostenlose Buchung".
