3
Architektur-Entscheidungen
EntscheidungUmsetzungBegründung
Kein Hamburger für unauthentifizierte UserHamburger-HTML wird nur gerendert wenn user vorhanden. _bindHamburger() bricht sofort ab wenn Element fehlt.Hamburger ohne Inhalt ist anti-pattern. Nur 2 Actions → keine Kollaps-Notwendigkeit.
Zwei separate Navbar-ButtonsNeues signupBtn-Option in Navbar.init() neben bestehendem loginBtn. Beide optional.Standard-Pattern (Airbnb, Preply): sekundäre CTA + primäre CTA. Klare visuelle Hierarchie.
„Anmelden" = ghost, „Registrieren" = secondary.btn.btn-ghost vs .btn.btn-secondaryPrimäre Conversion-Action (Registrieren) bekommt mehr Gewicht.
Tab-Switch Links im ModalInline <button> mit auth-switch-link Klasse unter dem Submit-Button jedes Panels.Eliminiert Verwirrung „Wo ist Registrieren wenn ich Sign-In sehe?"
AuthModal bleibt ein Modal (keine eigene Seite)UnverändertKein Page-Navigation-Interrupt beim Onboarding.
Sign-Up funktional für SchülerBereits implementiert — AppService.createUser()Lehrer bekommen Disziplin-Auswahl, Schüler werden direkt weitergeleitet.

5
Navbar.init() — API (vollständig)
navbar.js — Navbar.init(page, options)
/* Unauthentifizierte Seite (Landing) */ Navbar.init('landing', { loginBtn: { label: 'Anmelden', /* → .btn.btn-ghost links vom signupBtn */ onClick: function() { AuthModal.open('signin'); } }, signupBtn: { label: 'Registrieren', /* → .btn.btn-secondary rechts, primär */ onClick: function() { AuthModal.open('signup'); } } }); /* Authentifizierte Seite (teacher, student, dashboard…) */ Navbar.init('dashboard'); /* → Wallet-Badge + Hamburger-Drawer — loginBtn/signupBtn ignoriert */
Abwärtskompatibel. Alle bestehenden Seiten die Navbar.init('pagename') ohne Options aufrufen sind unverändert. Die neuen loginBtn/signupBtn-Options werden nur ausgewertet wenn kein eingeloggter User vorhanden ist (!user).
