Auto-Blacklist-Promotion — Implementierungslogik
Trigger: erste Buchung eines Schülers bei einem Lehrer
Hook in app-service.js → bookSlotWithEscrowSilent() — nach erfolgreichem Speichern
1
Prüfen: Ist dies die erste Buchung?
Nach dem Speichern zählen wir alle booked-Slots des Schülers bei diesem Lehrer. Wenn genau 1 → erste Buchung.
var allBooked = Store.Slots.byStudent(studentId);
var withTeacher = allBooked.filter(function(s) {
  return s.teacherId === teacherId && s.status === 'booked';
});
var isFirstBooking = (withTeacher.length === 1);
↓
2
Alle Slots des Lehrers scannen
Nur Slots mit new-only, blacklist-new oder excludeNewStudents: true sind betroffen.
var teacherSlots = Store.Slots.byTeacher(teacherId);
var toPromote = teacherSlots.filter(function(s) {
  return s.visibility === 'new-only'
    || s.visibility === 'blacklist-new'
    || s.excludeNewStudents === true;
});
↓
3
Schüler in visibilityList (Blacklist) eintragen
Für jeden betroffenen Slot: Schüler-UID zur visibilityList hinzufügen, visibility auf 'blacklist' setzen (wenn noch new-only). Bereits in Liste → skip. Flag autoPromoted: true setzen für UI-Kennzeichnung.
toPromote.forEach(function(slot) {
  var list = (slot.visibilityList || []).slice();
  if (list.indexOf(studentId) !== -1) return; /* bereits drin */
  list.push(studentId);
  var newVis = slot.visibility === 'new-only' ? 'blacklist' : slot.visibility;
  Store.Slots.update(slot.slotId, {
    visibility: newVis,
    visibilityList: list,
    autoPromotedStudents: (slot.autoPromotedStudents || []).concat([studentId])
  });
});
↓
✓
Ergebnis
Der Schüler ist jetzt fest in der Blacklist der betroffenen Slots. _isNew() wird für diesen Schüler irrelevant — die explizite Blacklist-Regel greift immer. Sichtbarkeit bleibt stabil, egal wie viele Buchungen noch kommen.

Das neue Feld autoPromotedStudents[] ermöglicht im Student Visibility View die Kennzeichnung als „Auto-BL" und die Override-Möglichkeit.
