Timezone Option B — Full Impact Assessment

Store all times as UTC minutes-from-midnight · 2026-03-24

1 — What "Store as UTC" means concretely

Current: slot.time = "09:00" (naive local string, assumed teacher TZ)

Option B: slot.utcMinutes = 480 — minutes since UTC midnight (09:00 UTC = 480)

Or alternatively: slot.time = "09:00" always means UTC, teacher's profile TZ determines display conversion.

Key insight: The format of "HH:MM" can stay — we just redefine its meaning from "teacher local" to "UTC". All arithmetic still works. Only the display layer changes.

3 — The DST problem with recurring slots

⚠ Recurring rules need special handling

A teacher sets a recurring slot: "Every Monday at 09:00 my local time (Europe/Berlin)".

In winter (UTC+1): stored as "08:00" UTC. In summer (UTC+2): stored as "07:00" UTC.

Solution: Recurring rules always store teacher-local time ("09:00"). When materialiseWeek() creates actual slots, it converts local → UTC using the teacher's timezone + the specific date (to get correct DST offset). Materialised slot stores UTC. This is correct and standard.

5 — Migration of existing data

⚠ Existing slots have no timezone context

All current slots store naive "HH:MM" — we don't know which timezone they were created in.

Migration strategy: On first launch after update, run a one-time migration:

  • For each existing slot: read teacher's current profile timezone (or browser TZ as fallback)
  • Convert slot.time from that TZ → UTC and overwrite
  • Mark migration complete with a version flag in localStorage

This is imperfect (if teacher already changed TZ before migration), but it's a one-time operation and the best we can do for historical data.
