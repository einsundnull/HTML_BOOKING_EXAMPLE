Navbar Identity Leak — Profile View
Beim Öffnen eines Lehrerprofils aus dem Katalog zeigte die Navbar den Kontostand und das vollständige Menü des Profil-Inhabers — nicht das des eingeloggten Betrachters. Root Cause: Navbar.init rief Auth.current() auf, das auf profile-view.html den falschen ?uid=-Parameter las.
🔴 Security · Kritisch 📅 21. März 2026 2 Dateien geändert · navbar.js · profile-view.html ✅ Behoben · APP_VERSION 2026-03-21-6
1

Root Cause

Kern des Problems: Navbar.init ruft intern immer Auth.current() auf. Auth.current() liest ?uid= aus der URL. Auf profile-view.html ist ?uid= jedoch der Profilinhaber (der Lehrer) — nicht der Besucher. Die Navbar renderte dadurch das vollständige Lehrer-Interface: Kontostand, Wallet-Badge, Hamburger-Menü mit Kalender, Finanzplaner etc. — für jeden Besucher des Profils.
Verwandter Bug (bereits behoben, BUGFIX #1): Derselbe Auth.current()-Fehler führte dazu, dass auch ProfileView den Profilinhaber als _viewer setzte und den Edit-Button für alle sichtbar machte. Beide Bugs haben dieselbe Root Cause — unterschiedliche Manifestationen.
❌ Vorher — Was angezeigt wurde
  • Wallet-Badge mit dem Kontostand des Lehrers
  • Hamburger-Menü mit Lehrer-Links (Kalender, Wallet, Finanzplaner)
  • Avatar des Lehrers in der Navbar
  • Alle Navbar-Links mit ?uid=t1 des Lehrers
  • Edit-FAB und „Profil bearbeiten"-Button sichtbar
✅ Nachher — Was angezeigt wird
  • Eingeloggt als Schüler: Schüler-Wallet, Schüler-Menü
  • Eingeloggt als Lehrer (eigenes Profil): eigenes Wallet, eigenes Menü
  • Nicht eingeloggt / öffentlich: nur „Anmelden"-Button, kein Wallet, kein Hamburger
  • Edit-FAB nur für Profilinhaber sichtbar

Implementierter Fix

2 Dateien · navbar.js · profile-view.html
Fix-Strategie: Navbar.init erhält eine neue Option viewerUid. Wenn gesetzt, wird dieser Wert für die Session-User-Auflösung verwendet statt Auth.current(). Alle bestehenden Aufrufer ohne viewerUid sind vollständig abwärtskompatibel — sie verhalten sich exakt wie bisher. Nur profile-view.html übergibt die neue Option.
Datei Stelle Änderung
js/navbar.js Navbar.init() — Zeile 67 Neue Option options.viewerUid. Wenn gesetzt: user = AppService.getUserSync(viewerUid). Wenn nicht gesetzt: user = Auth.current() (unverändertes Verhalten).
profile-view.html Inline-Script _init() Navbar.init('profile', navOptions) mit rollengerechtem navOptions: viewerUid wenn eingeloggt, sonst loginBtn für öffentliche Ansicht.
js/app-config.js APP_VERSION '2026-03-21-5' → '2026-03-21-6'
navbar.js — Navbar.init() — neue viewerUid Option
/* options (optional): { loginBtn: { label, onClick }, signupBtn: { label, onClick }, viewerUid: string — explicit session-user uid. Use on pages where ?uid= identifies the content being viewed (not the session owner). When omitted, Auth.current() (reads ?uid= from URL) is used. } */ init: function(activePage, options) { var self = this; self._options = options || {}; /* Resolve session user: If options.viewerUid is supplied use that — prevents profile-view.html from mistakenly treating the profile owner as the logged-in user. */ var user; if (self._options.viewerUid) { user = AppService.getUserSync(decodeURIComponent(self._options.viewerUid)) || null; } else { user = Auth.current(); } self._user = user; // ... rest unchanged
profile-view.html — _init() — viewerUid übergeben
function _init() { /* Build Navbar with correct session context: - viewerUid present → viewer is logged in → show their wallet + menu - viewerUid absent → public / unauthenticated → show login button only Never pass Auth.current() here — on profile-view.html ?uid= identifies the PROFILE being viewed, not the session owner. */ var navOptions = viewerUid ? { viewerUid: viewerUid } : { loginBtn: { label: 'Anmelden', onClick: function() { window.location.href = './landing.html'; }}}; Navbar.init('profile', navOptions); ProfileView.init(); if (viewerUid && profileUid && viewerUid !== profileUid) { ChatPanel.init({ uid: viewerUid }); } }

Abwärtskompatibilität — alle Navbar.init Aufrufer

Kein bestehender Aufrufer muss geändert werden. Die viewerUid-Option ist additiv — kein bestehender Code übergibt sie, alle fallen in den unveränderten Auth.current()-Pfad.
