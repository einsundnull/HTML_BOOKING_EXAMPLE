6 — Implementierungsplan (4 Phasen)
Phase 1 — Slot-Schema + Booking-Record (store.js + adapter-localstorage.js) · ~1h
  1. store.js: bookSlot() setzt bookedAt: isoNow() Datenbasis für alle nachfolgenden TX
  2. store.js: cancelBooking() setzt cancelledAt + cancelledBy Wer hat wann storniert
  3. adapter-localstorage.js: neue Funktion writeBookingRecord() — schreibt booking-TX beim Schüler + Lehrer Auch bei cash_on_site ohne Escrow
  4. bookSlotWithEscrow() in AppService ruft writeBookingRecord() nach erfolgreichem Book auf Zentraler Einstiegspunkt
Phase 2 — Move-Record + Teacher-Cancel-TX + Lesson-Confirm-TX (adapter + AppService) · ~1.5h
  1. adapter-localstorage.js: neue Funktion writeMoveRecord(oldSlot, newSlot, initiatorId, role, cb) Schreibt move-TX mit vorher/nachher
  2. AppService: moveSlotWithRecord(oldSlotId, newSlotId, initiatorId, role, cb) — kombiniert cancel + book + writeMove Atomic Move-Operation
  3. ui.js confirmBtn: ersetzt direkte pending-Map-Manipulation durch AppService.moveSlotWithRecord() Move wird jetzt lückenlos geloggt
  4. cancelSlotWithPolicy(): bei teacher_cancel-Tier zusätzliche TX beim Lehrer schreiben Wer hat wann storniert — aus Lehrer-Sicht
  5. confirmLesson(): zusätzliche lesson_confirmed-TX beim Schüler Stunde bestätigt in Schüler-History
Phase 3 — Admin getAllTransactions + Filter-API (adapter + AppService) · ~1h
  1. adapter-localstorage.js: getAllTransactions(opts, cb) mit opts: { uid, role, types[], dateFrom, dateTo, teacherId, studentId, counterpartId }
  2. app-service.js: getAllTransactions(opts, cb) und getBookingHistory(uid, opts, cb) exportieren
  3. adapter-firestore.js: Stubs für beide neuen Funktionen
Phase 4 — Admin Audit Screen (admin.html + admin.js) · ~3h
  1. Neuer Tab "Audit Log" in admin.html (neben bestehenden Tabs) mit Tabelle + Filter-Bar
  2. admin.js: renderAuditLog() — lädt via AppService.getAllTransactions(), rendert gefilterte Tabelle
  3. Filter-Bar: Freitext (Name/UID), Rolle, Typ, Zeitraum, Stundendatum, Gegenpartei
  4. Aufklapp-Zeile pro TX mit allen Meta-Feldern
  5. Export-Button: CSV-Download der gefilterten Ergebnisse
7 — Hinweise zur Datensicherheit und Firestore-Migration
PunktBeschreibung
Unveränderlichkeit TXs werden nur hinzugefügt (prepend), nie bearbeitet. app_transactions ist ein append-only Log. Bei Firestore: eigene Collection /transactions ohne Update-Rechte.
Balance-Validierung Jede TX speichert balance = Saldo nach TX. Admin kann Konsistenz prüfen: Summe aller amount eines Users muss balance des letzten TX ergeben.
Relationale Integrität relatedUid in jeder TX zeigt auf die Gegenpartei. Beide Seiten (Student + Teacher) bekommen bei finanziellen Ereignissen eine eigene TX. Dies ermöglicht vollständige Sicht für beide.
Firestore-Kompatibilität Alle neuen Funktionen folgen dem bestehenden Callback-Muster. Firestore-Stubs werden parallel angelegt. localStorage-Keys bleiben kompatibel.
