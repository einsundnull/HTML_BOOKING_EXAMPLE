1 — Grundprinzip: UTC-Anchored Absolute Time

Das System verwendet das UTC-anchored Absolute Time Model (auch: Fixed Time Model). Alle Zeiten werden intern als UTC gespeichert und sind timezone-unabhängig. Die Anzeige wird zur Laufzeit in die Lokalzeit des jeweiligen Betrachters konvertiert — niemals die gespeicherte Zeit selbst.

Kernregel: Eine gespeicherte Zeit ändert sich nie. Nur die Anzeige ändert sich je nach Timezone des Betrachters. Das ist identisch zum Verhalten von Kalender-Systemen wie Google Calendar (bei „specific time"-Events).

Beispiel

AktionUTC (gespeichert)Teacher sieht (Georgia UTC+4)Student sieht (Berlin UTC+1)
Teacher in Georgia setzt Slot 08:30 UTC 12:30 (lokal) 09:30 (lokal)
Teacher zieht nach Berlin, stellt TZ um 08:30 UTC — unverändert 09:30 (lokal Berlin) 09:30 (lokal) — unverändert
Student ändert seine TZ von Berlin auf Tokyo (UTC+9) 08:30 UTC — unverändert 09:30 (lokal Berlin) — unverändert 17:30 (lokal Tokyo)

/* DST-korrekter Offset-Trick — kein Moment.js, keine externe Lib */
function _getOffsetMinutes(ianaTimezone, dateStr) {
  var base = (dateStr || '2026-01-01') + 'T12:00:00';
  var d   = new Date(base);
  var utc = new Date(d.toLocaleString('en-US', { timeZone: 'UTC' }));
  var loc = new Date(d.toLocaleString('en-US', { timeZone: ianaTimezone }));
  return (loc - utc) / 60000; /* Minuten, positiv = östlich von UTC */
}

createSlot: _utc-Flag

createSlot konvertiert standardmäßig die übergebene Zeit via localToUtc. Caller die bereits UTC übergeben, setzen _utc: true um die Konvertierung zu überspringen. Das Flag wird nicht persistiert.

/* Caller mit lokaler Zeit (z.B. Drag & Drop, Formular-Eingabe): */
AppService.createSlot({ teacherId, date, time: '12:30', status: 'available' });
/* → adapter konvertiert: localToUtc('12:30', date, teacherTZ) → speichert UTC */

/* Caller mit UTC-Zeit (Week Grid, Day View, flushGridPending): */
AppService.createSlot({ teacherId, date, time: '08:30', _utc: true, status: 'available' });
/* → adapter überspringt Konvertierung, speichert '08:30' direkt als UTC */
5 — Grid-Aufbau: _buildLocalOrderedUtcTimes (ui.js)

Gemeinsame Funktion für Day View, Teacher Week Grid und Student Week Grid. Gibt 48 UTC-Strings zurück, geordnet so dass Index 0 = lokale 00:00 entspricht. Die Zeilen-Labels sind immer "00:00" bis "23:30" lokal — die UTC-Werte darunter verschieben sich je nach Timezone.

/* ui.js — window._buildLocalOrderedUtcTimes */
function _buildLocalOrderedUtcTimes(dateStr, uid) {
  var offsetMin = TimezoneService.getOffsetMinutes(
    TimezoneService.getUserTimezone(uid), dateStr
  );
  /* UTC-Minuten von lokalem Mitternacht: */
  var midnightUtcMin = ((-offsetMin) % 1440 + 1440) % 1440;

  var times = [];
  for (var i = 0; i < 48; i++) {
    var m = (midnightUtcMin + i * 30) % 1440;
    times.push(pad(Math.floor(m/60)) + ':' + pad(m%60));
  }
  return times; /* times[0] = lokale 00:00 in UTC */
}

DST-Übergangswoche: Auf einer Woche die eine Zeitumstellung enthält (z.B. 29. März 2026, Europa UTC+1→UTC+2) hat jede Spalte einen anderen UTC-Offset. Deshalb berechnet das Grid localToUtc(localLabel, cellDateStr) pro Zelle — nicht einmalig für die ganze Woche. So findet Samstag und Sonntag der gleiche lokale Label "12:30" jeweils den richtigen UTC-Slot.
6 — materialiseWeek (adapter-localstorage.js)

Materialisiert Recurring Rules für eine bestimmte Woche in konkrete Slots. Da rule.time UTC ist, wird keine Konvertierung benötigt — die Zeit wird direkt als Slot-Zeit übernommen.

/* adapter-localstorage.js */
function materialiseWeek(teacherId, weekDates, callback) {
  var rules = Store.Recurring.byTeacher(teacherId);

  /* 1. PURGE — alle nicht-gebuchten available-Slots dieser Woche löschen.
     Verhindert Duplikate bei wiederholtem Aufruf oder nach Datenkorruption. */
  var purged = allSlots.filter(function(s) {
    if (s.teacherId !== teacherId)                    return true; /* anderer Teacher */
    if (!weekDateStrs[s.date])                         return true; /* andere Woche */
    if (s.studentId || s.confirmedAt || s.bookedAt)    return true; /* gebucht */
    if (s.status === 'timeout')                       return true; /* Timeout */
    return false; /* plain available → löschen */
  });

  /* 2. MATERIALISE — für jede Rule + Wochentag einen Slot anlegen */
  rules.forEach(function(rule) {
    var dateStr = weekDates[rule.dayOfWeek].toISOString().slice(0,10);
    /* rule.time ist UTC — direkt verwenden, keine Konvertierung */
    var key = teacherId + '|' + dateStr + '|' + rule.time;
    if (!existSet[key]) {
      purged.push({ date: dateStr, time: rule.time, status: 'available', ... });
    }
  });
}

8 — Verhalten bei Timezone-Wechsel
Wer ändert TZ?Was passiert?Was bleibt gleich?
Teacher ändert Timezone im Profil Grid und Day View zeigen Slots zu anderen lokalen Uhrzeiten. Recurring Rule-Zeilen verschieben sich im Grid. UTC-Wert der Slots und Rules. Student sieht keine Änderung.
Student ändert Timezone im Profil Alle Slots im Student-Grid und in der Buchungshistorie verschieben sich um das Delta. Teacher sieht keine Änderung. UTC-Wert der Slots. Teacher-Grid unverändert.
Sommerzeitumstellung (DST) In der Übergangswoche berechnet jede Grid-Zelle ihren UTC-Key per-Datum via localToUtc(localLabel, cellDate). Slots bleiben an der richtigen lokalen Position. UTC-Wert der Slots. Lokale Anzeige korrekt dank per-Zelle-Konvertierung.
Achtung bei TZ-Änderung des Teachers: Bestehende Recurring Rules behalten ihren UTC-Wert. Das bedeutet: der Teacher sieht die Slots nach TZ-Wechsel zu anderen Lokalzeiten. Er muss ggf. die Recurring Rules neu setzen, wenn er seine Verfügbarkeit an die neue lokale Zeit anpassen will. Der Student sieht in jedem Fall keine Änderung.
