Timeout-Logik — Root Cause Analysis

Warum klicken auf "Timeout" keine sichtbare Änderung erzeugt.

# File Status Problem Fix
Bug 1 — Optimistic Update liest alten Store-Wert
1 teacher.js ROOT CAUSE Der Timeout-Button macht:
1. _tdvRenderSlots() ← sofort
2. AppService.applySlotTimeout(...) ← danach

_tdvRenderSlots() liest AppService.getSlotsByTeacherDateSync() — zum Zeitpunkt des Aufrufs ist der Store noch nicht aktualisiert. Der Slot hat noch status='available'. Die UI zeigt keine Änderung. 		Reihenfolge umkehren: zuerst speichern, dann rendern.
Kein echtes Optimistic Update möglich solange der Adapter synchron arbeitet und getSlotsByTeacherDateSync() direkt aus dem Store liest.

Fix: applySlotTimeout zuerst aufrufen, _tdvRenderSlots() im Callback.
Bug 2 — Status-Ermittlung in _tdvBuildRow liest baseStatus statt status
2 teacher.js BUG In _tdvBuildRow():
var status = slot ? (slot.baseStatus || slot.status) : 'empty';

slot.baseStatus wird zuerst gelesen. Nach applyTimeout ist status='timeout' aber baseStatus='available' — also zeigt die Row weiterhin "Verfügbar", weil baseStatus Vorrang hat. 		Richtige Reihenfolge: zuerst slot.status lesen, baseStatus nur als Fallback wenn status undefined ist:
var status = slot ? (slot.status || slot.baseStatus) : 'empty';
Bug 3 — rowClass für timeout-avail nie gesetzt (wegen Bug 2)
3 teacher.js CONSEQUENCE Da Bug 2 status immer als 'available' liefert wenn baseStatus='available', kommt der Code
rowClass += 'dv-s-timeout-avail'
nie zur Ausführung → kein gelber Hintergrund. 		Wird automatisch behoben wenn Bug 2 gefixt ist.
Call Chain — Vollständig verifiziert
— teacher.js OK toBtn2.click → AppService.applySlotTimeout(slotId, cb) Aufruf korrekt
— app-service.js OK _call('applySlotTimeout', [slotId, cb]) → findet Methode im Adapter Methode existiert, kein Fehler
— adapter-localstorage.js OK Store.Slots.applyTimeout(slotId) Methode existiert, korrekt exportiert
— store.js OK Setzt status='timeout', lässt baseStatus unverändert, speichert Logik korrekt
Summary — 2 Fixes in teacher.js
Fix 1 teacher.js Alle Aktionen: _tdvRenderSlots() NACH dem Adapter-Call im Callback ausführen, nicht davor.
LocalStorageAdapter ist synchron — Callback wird sofort aufgerufen, also ist "optimistic" hier dasselbe wie "nach dem Speichern".
Fix 2 teacher.js Zeile in _tdvBuildRow() ändern:
var status = slot ? (slot.baseStatus || slot.status) : 'empty';
→
var status = slot ? (slot.status || slot.baseStatus) : 'empty';
